## Deviations + justifications

1. **`NSMotionUsageDescription` included (spec said "optionally").** Added because HealthKit step-count queries surface motion/step data and iOS will reject reads without the usage string. Low-risk, additive.
2. **`eas.json` change = `prebuildCommand: "prebuild --no-install"` on all three profiles.** Config plugins already apply automatically during managed prebuild, so the build profiles "support native modules" as-is. To make the requirement explicit and deterministic without altering existing behavior, the documented EAS `prebuildCommand` field was added; `--no-install` matches EAS's managed flow (EAS runs its own dependency install). Behavior-preserving.
3. **No separate Samsung Health npm bridge installed.** Modern Samsung Health syncs into Android Health Connect, which is already covered by `react-native-health-connect` + the declared `android.permission.health.*` permissions. The Samsung-specific Sensor SDK permission (`READ_ADDITIONAL_HEALTH_DATA`) was declared so PR-HK-2.c has everything it needs. No additional config PR required.
4. **`package-lock.json` shows 12 deletions.** These are `dev`/`devOptional` flag reclassifications npm makes when previously dev-only transitive packages become production deps via the new runtime modules. No package was removed; lockfile is additive in substance.
