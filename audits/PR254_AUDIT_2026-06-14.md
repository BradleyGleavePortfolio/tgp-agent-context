## Per-finding detail

### F1 (P3) — Stale docstring: "Polled on Coach Home focus" with no focus-refresh mechanism

**File:** `src/hooks/useCoachThreeArcCounts.ts:5`

```ts
/**
 * useCoachThreeArcCounts — TanStack Query hook for the ED.2 three-arc router.
 *
 * Reads GET /coach/home/daily-rings (the calling coach's three completion arcs
 * for today). Polled on Coach Home focus; the backend memoises for 30s, so the
 * client mirrors that with a 30s staleTime to avoid hammering the endpoint.
 */
```

The phrase "Polled on Coach Home focus" implies a `useFocusEffect`-driven invalidation or a `useIsFocused`-gated `refetchInterval` — the pattern used by `AIBudgetMount` (which wraps `useIsFocused()` explicitly for the same reason). Neither exists here. The global `queryClient` (`src/services/queryClient.ts:48`) sets `refetchOnWindowFocus: false` for the whole app ("Mobile-first: do not auto-refetch on window focus. RN re-mounts on focus already and we don't want a screen tab swipe to fire a fan-out."). There is no `refetchInterval`, no `useFocusEffect`, and no `useIsFocused` call in the hook or in `CoachHomeScreen`'s usage of it.

The actual behavior is: data fetches once on mount, then refetches on the next mount where data is stale (> 30 s). A coach who completes check-ins, navigates to the client list, and returns immediately (within 30 s) will see unchanged counts until the staleness window expires and they navigate away-and-back. Given the backend's own 30 s cache, the effective maximum drift is reasonable and the design is arguably correct — but the docstring describes a stronger guarantee ("polled on focus") that the code does not deliver. A future developer could act on the docstring and add a focus-refresh expecting it was always there.

P3 (doc inaccuracy; behavior is safe and intentional).

**Recommended fix:** Reword the first paragraph of the hook docstring to match reality: "Fetches once on mount; data is considered stale after 30 s (mirroring the backend's server-side cache) and will be refetched on the next mount when stale." If focus-on-navigate-back freshness is desired, add `useFocusEffect` + `queryClient.invalidateQueries(coachThreeArcCountsKeys.all)` in `CoachHomeScreen` (mirroring `AIBudgetMount`'s `useIsFocused` pattern).

### F2 (P3) — `accessibilityState={{ busy: false }}` hardcoded; not wired to query loading state

**File:** `src/components/coach/CoachThreeArcRouter.tsx:206-207`

```tsx
<View
  style={styles.wrap}
  testID={testID}
  accessibilityRole="summary"
  accessibilityState={{ busy: false }}   // ← hardcoded
>
```

The component receives `rings?: DailyRings` (undefined while `dailyRingsQuery.isPending` in the host). When `rings` is undefined the arcs render zeroed — a correct graceful degradation — but `busy: false` is reported to the AT throughout, including during the initial network fetch. Assistive-technology users who navigate to Coach Home on a cold start receive no loading signal from this element; VoiceOver/TalkBack reads "summary, 0/0, 0 percent" rather than indicating that fresh data is arriving.

The component has no `isLoading` prop, so the fix requires either adding one or removing the `accessibilityState` entirely (the element's children convey state adequately through their own labels and `accessibilityLiveRegion="polite"` on the voice line).

P3 (minor a11y gap on an element that already communicates its state through child labels; not a crash or data-loss path; flag is OFF in production).

**Recommended fix:** Either (a) add an `isLoading?: boolean` prop to `CoachThreeArcRouterProps` and set `accessibilityState={{ busy: isLoading ?? false }}`, with the host passing `rings={dailyRingsQuery.data} isLoading={dailyRingsQuery.isPending}`, or (b) remove the `accessibilityState` prop entirely and rely on the `accessibilityLiveRegion="polite"` voice-line announcement once data lands.

- **AbortSignal on network read.** `AbortSignal.timeout(DAILY_RINGS_REQUEST_TIMEOUT_MS)` (15 s) in `coachDailyRingsApi.get()`. Consistent with the community API pattern.

- **R0 / quiet-luxury clean.** No `Coming soon`, `@ts-ignore`, `.catch(()=>undefined)`, `as unknown as`, `as any` in any prod file. No `TODO`/`FIXME`/`XXX`. No `fontWeight: '700'|'800'` in any introduced line.
- **No `mutedGold` fill.** Arc fill uses `colors.accent` (oxblood / `#4A0404` light, `#B43C3C` dark) — not `mutedGold`. PR #242 F3 lesson applied.
- **Token-only colors.** All colors resolve through `useTheme().semanticColors` (`accent`, `border`, `textMuted`, `textPrimary`) — all confirmed present in both `lightTokens` and `darkTokens`. No raw hex.
- **Touch targets.** `arcPressable` content height = ARC_SIZE (72) + `spacing.sm` (8) gap + label (~20) ≈ 100 dp; width = ARC_SIZE (72). Well above the 44/48 dp floor.
- **Per-arc accessibility.** Each arc is `accessibilityRole="button"` with a descriptive label (`"CHECK-INS, 3/5, 60 percent"`) and a contextual `accessibilityHint`. Voice-line has `accessibilityLiveRegion="polite"`. Test pins the button role and label content.
- **RNTL v14.** Every `render()` in both test files is `await`-ed.
- **No Reanimated / no `useNativeDriver: false`.** Arcs are static SVG (`strokeDashoffset` from fraction). `FadeInView` wrapper uses `useNativeDriver: true` (pre-existing). No motion to gate; reduce-motion posture is correctly noted in the component header.
- **Single caller.** `CoachThreeArcRouter` and `coachDailyRingsApi` are called exclusively from `CoachHomeScreen` (confirmed by grep). No surprise callers.
- **`useTheme().semanticColors` (Phase 11 tokens).** Correct alias — matches `ThemeProvider`'s `semanticColors: SemanticTokens` field. No `semanticColors.surface` mistake.
- **All coaches, not just owners.** Mount not gated on `isOwner`; backend `@Roles('coach')` covers both coach and owner — correct.
- **Deep-links to real routes.** All three `onPress` callbacks navigate to real, registered stack screens (`ClientsStack`, `SettingsStack > CoachBrief`, `Messages`). Static pin test asserts the exact route strings.
