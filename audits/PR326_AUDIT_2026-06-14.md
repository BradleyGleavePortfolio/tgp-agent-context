## Verdict

**CHANGES_REQUESTED**

One P1 and two P2 findings. No P0. P0: 0 · P1: 1 · P2: 2 · P3: 1.

The P1 is a comment/code divergence in the core update loop: the code promises to re-assert `status='pending'` in the `WHERE` clause of each per-drop `update` to defeat a concurrent dispatcher claim racing mid-transaction, but the actual `where` object is `{ id: drop.id }` only. Under Prisma's default `READ COMMITTED` isolation (no explicit `isolationLevel` on this `$transaction`), the dispatcher's `UPDATE scheduledDrop SET status='dispatching' WHERE id=? AND status='pending'` can commit between the inner `findMany` and the `update`, causing `pushToExisting` to silently mutate a row the dispatcher has already claimed. This corrupts the dispatcher's in-flight snapshot (cadence / fire_at fields are overwritten on an actively-dispatching row). No production exposure today because the endpoint is behind the `coach`/`owner` roles gate and requires deliberate invocation, but the defect is real and reachable on any concurrent push+cron tick. The P2s are: (1) missing `@Throttle` on the new write endpoint (the only write route on this controller without an explicit bucket), and (2) missing Zod response schema (`drops_updated, buyers_affected, skipped_delivered` is a plain TypeScript interface with no runtime `.parse()` fence on the way out — inconsistent with the repo's Zod `.strict()` response doctrine). The P3 is the lack of an `AuditService` entry for a bulk-mutating coach action.

## Per-finding detail

### F1 (P1) — dispatcher-claim race: `update WHERE` lacks `status:'pending'`

**File:** `src/packages/package-contents.service.ts:647`

```ts
await tx.scheduledDrop.update({
  // Re-assert status='pending' in the WHERE so a sibling
  // dispatcher claim mid-tx can never flip a row we believed
  // was pending. With this composite filter Prisma's update
  // throws P2025 when the row was claimed under us, rolling
  // the whole push back — the dispatch wins, the push retries
  // from a clean slate next call. Use updateMany so the
  // composite filter is honored; count===1 indicates success,
  // count===0 means the row was claimed under us and we
  // must roll the tx back so the partial push doesn't leak.
  where: { id: drop.id },          // ← status: 'pending' is ABSENT
  data: {
    display_title: liveContent.display_title,
    ...
    fire_at: newFireAt,
  },
});
dropsUpdated += 1;                  // ← always incremented even on a claimed row
```

The comment is the spec; the implementation doesn't satisfy it. The codebase's `drip-dispatcher.cron.ts` performs an atomic claim via:

```ts
await this.prisma.scheduledDrop.updateMany({
  where: { id, status: priorStatus, materialised_ref: null, ... },
  data: { status: 'dispatching', locked_at: now },
});
```

Under `READ COMMITTED` (Prisma's default; no explicit `isolationLevel` is set on this `$transaction`): between the inner `findMany({ where: { content_id, status: 'pending', id: { in: pendingIds } } })` and the per-row `update`, the dispatcher can commit `status='dispatching'` on a row. The `update({ where: { id: drop.id } })` then succeeds — Prisma `update` requires the row to exist by the WHERE key, not that it matches any additional filter — overwriting `cadence_kind`, `cadence_payload`, and `fire_at` on a row the dispatcher has already locked for delivery. The dispatcher's `dispatch()` then materialises content using now-stale snapshot fields.

Additionally, the per-row `dropsUpdated += 1` fires unconditionally, so the returned `drops_updated` count overcounts any rows that were actually dispatching at update time.

The advisory lock (`pg_advisory_xact_lock(NAMESPACE, hashtext(packageId))`) serializes concurrent `pushToExisting` or `attach/reorder/patch` calls on the same package, but the dispatcher does **not** acquire this lock, so the lock provides no protection against the push↔dispatcher race.

**50-Failures reference:** #28 (race-safe check-and-set) and #44 (rollback safety — the partial-push-without-status-guard leaks an inconsistent update).

**Recommended fix:** Switch to `updateMany` with `where: { id: drop.id, status: 'pending' }` and check `count === 0` as a contention signal:

```ts
const updated = await tx.scheduledDrop.updateMany({
  where: { id: drop.id, status: 'pending' },
  data: { ... },
});
if (updated.count === 0) {
  // Row was claimed by dispatcher between our read and this write.
  // Rolling back the whole tx (throw) is the safe choice per the comment's own spec:
  throw new ConflictException({
    error: 'DROP_CLAIMED_UNDER_PUSH',
    message: `drop ${drop.id} was claimed by the dispatcher during push; retry`,
  });
}
dropsUpdated += 1;
```

Alternatively, raise the isolation level to `REPEATABLE READ` so the inner `findMany` result is stable for the lifetime of the transaction, making a concurrent status flip visible as a serialization error.

### F2 (P2) — missing `@Throttle` on bulk-write endpoint

**File:** `src/packages/package-contents.controller.ts:127`

```ts
@Roles('coach', 'owner')
@Post(':contentId/push-to-existing')
@HttpCode(HttpStatus.OK)
async pushToExistingDrops(...) { ... }
```

No `@Throttle` decorator. Every other write route on this controller (`@Post()` attach, `@Put('reorder')`, `@Patch(':contentId')`, `@Delete(':contentId')`) is also unthrottled — that pre-existing gap is out of this PR's scope. However, this new endpoint performs the most expensive write of all: up to 10 000 per-row UPDATEs inside a locked `$transaction`. Community write routes (`community-classroom.controller.ts:58–135`) all carry explicit `@Throttle({ default: { ttl: 60_000, limit: COMMUNITY_POSTS_PER_MIN } })`. The absence of a named bucket here means a coach can hammer this endpoint at 300 req/min (the global authed default), chaining 10k-row push transactions back-to-back.

**Recommended fix:** Add an explicit `@Throttle` write bucket consistent with the repo's mutation rate limits (e.g. `{ default: { ttl: 60_000, limit: 10 } }` — one push per 6 seconds per user is a reasonable ceiling for a bulk action that touches buyers' schedules).

### F3 (P2) — no Zod `.strict()` response schema

**File:** `src/packages/package-contents.service.ts:466–476` / `src/packages/package-contents.dto.ts`

The service return type is declared as an anonymous TypeScript interface:

```ts
async pushToExisting(...): Promise<{
  drops_updated: number;
  buyers_affected: number;
  skipped_delivered: number;
}> { ... }
```

No corresponding `PushToExistingResponseSchema = z.object({...}).strict()` exists in `package-contents.dto.ts`. The repo's own doctrine (established in the classroom PR #396 audit and the `main.ts` global pipe) requires all API response shapes to be Zod `.strict()` parsed on the way out. An unvalidated return type means any future addition of sensitive internal fields (e.g. buyer identifiers, content row internals) would silently flow to the API consumer without a compile-time or runtime catch.

**Recommended fix:**

```ts
export const PushToExistingResponseSchema = z.object({
  drops_updated: z.number().int().nonnegative(),
  buyers_affected: z.number().int().nonnegative(),
  skipped_delivered: z.number().int().nonnegative(),
}).strict();
export type PushToExistingResponse = z.infer<typeof PushToExistingResponseSchema>;
```

Parse with `.parse(result)` before returning from the service, and use `PushToExistingResponse` as the declared return type.

### F4 (P3) — no `AuditService` entry for bulk push

**File:** `src/packages/package-contents.service.ts:679`

```ts
this.logger.log(
  `pushToExisting: package=${packageId} content=${contentId} drops_updated=${result.drops_updated} ...`,
);
```

Structured log only; no `AuditService.record(...)` or equivalent durable audit-log write. A coach deliberately propagating a content edit to all buyers' pending drops is a high-value irreversible mutation (changing fire_at for potentially thousands of deliveries). This is at least as significant as the first-payment emit (PR #395 F8), which was cited P3 for the same gap.

**Recommended fix:** Add an `AuditService.record(...)` entry keyed on `(coachUserId, packageId, contentId, drops_updated, buyers_affected)` before returning, consistent with other high-value write paths in the codebase.

## Hectacorn bar

Would Stripe/Linear/Apple ship this bulk-mutation endpoint? **Not yet, due to F1.** The ownership model, sub-coach scope, advisory-lock serialization, defensive cap, per-buyer anchor computation, and atomic rollback are all architecturally sound and match the codebase's established patterns. The critical shortfall is F1: the comment documents the correct race-safety contract (`status='pending'` in the WHERE, P2025-on-claim as the rollback signal), but the implementation delivers `update({ where: { id } })` — a well-known check-and-set failure mode the codebase's own `drip-dispatcher.cron.ts` solves correctly via `updateMany({ where: { id, status: priorStatus, ... } })`. A Stripe-grade mutations team would catch this in code review before merge. The P2/P3 gaps (throttle, response schema, audit log) are polish-level gaps that don't break correctness but do reduce production observability and rate-protection.
