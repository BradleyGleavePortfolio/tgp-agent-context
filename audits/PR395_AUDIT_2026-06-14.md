## Findings

| ID | Sev | Area | Finding |
|----|-----|------|---------|
| F1 | **P0** | Webhook safety / dedup | Notification rows are written on `NotificationsService`'s own Prisma client, NOT the ambient `tx`. The emit fires before the outer tx commits and the rows commit immediately. On any post-emit rollback (e.g. fanout resolver throws — explicitly documented as re-throwing and rolling back) the ledger row + event-id dedup row roll back but the notification rows do NOT. Stripe redelivers → re-INSERT → emit fires AGAIN → **duplicate first-payment notification(s)**. This is the exact "double-emit causing duplicate notification" the P0 rubric names, and it defeats the PR's stated "exactly once, forever" + rollback-safety (50-Failures #44) guarantee. |
| F2 | **P1** | Transaction boundary | Root cause of F1, called separately because it also breaks the happy path: even with no rollback, the celebration push/inapp is dispatched *before commit*. If the outer `$transaction` commit itself fails, the coach receives a "first payment" notification for a purchase that never persisted. The emit must ride the purchase tx (thread `tx` into the emitter / write the notification via `tx.notification.create`). |
| F3 | **P2** | Spec ↔ impl mismatch | PR title says "first payment **per coach×client**" and the audit brief lists "second payment for same **coach×client** → NO emit". The implementation enforces `coachId @unique` = first payment **per coach, forever** (any later client is suppressed). The plan, schema comments, and tests are internally consistent on per-coach-forever, so this is a deliberate design — but the title/brief wording is wrong and will mislead. Needs the title/spec corrected (or the constraint changed) so intent is unambiguous. |
| F4 | **P2** | Missing edge-case test | No test for "two different coaches, same client → both emit" (brief item 5). Only same-coach cases are covered. |
| F5 | **P2** | Missing edge-case test + undefined behavior | Refund/chargeback after the first payment is undefined and undocumented: the ledger row is never reversed, so a fully-refunded first payment still permanently consumes the coach's once-ever celebration (a real first payment later never fires). No test, no documented decision. |
| F6 | **P2** | Missing edge-case test (Connect) | No test for the Stripe Connect / sub-coach path. The emit uses `purchase.coach_user_id` (the *selling* coach), which is a defensible attribution, but the head-coach-vs-sub-coach behavior is unverified and undocumented against brief item 3. |
| F7 | **P3** | Test hygiene | `as unknown as` appears 3× in the new `.spec.ts` files (`coach-first-payment.service.spec.ts:56,155`, `first-payment-webhook.integration.spec.ts:56`). ## Evidence

### F1 (P0) — notification writes escape the purchase transaction → duplicate emit on retry
`coach-first-payment.service.ts` inserts the ledger row on the ambient `tx`, then calls the emitter with NO tx:
```
await tx.coachFirstPaymentNotification.create({ data: { coachId, amount, currency, clientId } });  // in tx
...
await this.emitter.emit(coachId, { amount, currency, clientId });   // <-- no tx threaded
```
`first-payment.emitter.ts` → `this.notifications.createNotification(...)` (×2: inapp + push). `notifications.service.ts:282` `createNotification` ends in:
```
return this.prisma.notification.create({ data: { ... } });   // NotificationsService's OWN PrismaService — autocommits, NOT the outer tx
```
The webhook callsite runs the emit, then continues IN the same outer tx with fanout that can re-throw:
```
await this.maybeEmitFirstPayment(updated, tx);            // notifications already committed here
...
await this.fanout.onPurchaseEntitled(updated, {...}, tx); // "A resolver failure here re-throws and rolls the whole event back; Stripe retries."
```
Rollback discards the ledger row + the `stripe_event_id` dedup row (both written inside `billing.service.ts` `$transaction`), so Stripe's redelivery re-processes from scratch, re-inserts the ledger row, and emits again → the coach receives 2+ first-payment notifications. The ledger's `coachId @unique` only makes the *row* exactly-once; it does NOT make the *user-visible notification* exactly-once once the rows live outside the tx.

### F2 (P1) — emit-before-commit
Same code path: the push/inapp dispatch is synchronous inside `tryEmitFirstPayment`, executed before `BillingService.prisma.$transaction(...)` commits. A commit failure (serialization error, connection drop) after the emit leaves a delivered celebration with no committed purchase. Fix = make the notification write transactional with the purchase, or move the emit to a post-commit hook keyed off the committed ledger row.

### F3 (P2) — per-coach-forever vs per-coach×client
`schema.prisma`: `coachId String @unique // <- DB enforces "exactly once"`. Migration header: "exactly once, forever ... Exactly one row per coach can ever exist." Service comment: "the coach's first-ever payment." All internally consistent = first payment per coach forever. But PR title = "per coach×client" and brief test #4 = "second payment for same coach×client → NO emit". The implementation suppresses a coach's payments from *all* later clients, not just the same client. Title/spec must be corrected to match the (deliberate) per-coach design, or the unique key changed to `@@unique([coachId, clientId])` if per-coach×client was truly intended.

### F4–F6 (P2) — test matrix gaps
`it(...)` titles present: first-payment-inserts-one-row, idempotent-twice-same-coach, second-payment-same-coach-no-dup, shares-caller-tx, rethrow-non-P2002; integration: flag-off (unset), flag-off (false), flag-on-server-trusted-fields, flag-on-no-tx. Absent: two-coaches-same-client (F4), refund/chargeback-after-first (F5), Stripe-Connect/sub-coach routing (F6). Brief item 5 explicitly requires all three.

### F5 (P2) — refund/chargeback undefined
`grep` of `refund-dispute-handler.service.ts` shows zero references to the first-payment ledger. A refunded/charged-back first payment is never un-recorded, permanently consuming the once-ever celebration. No documented decision either way.

### F7 (P3) — test-only `as unknown as`
`coach-first-payment.service.spec.ts:56` `{ emit } as unknown as FirstPaymentEmitter`; `:155` `(tx as unknown as {...})`; `first-payment-webhook.integration.spec.ts:56` `{ tryEmitFirstPayment } as unknown as CoachFirstPaymentService`. Test files only; production `coach-first-payment.service.ts` and `first-payment.emitter.ts` are clean.

## What's CORRECTLY implemented

- **Server-trusted inputs (IDOR / 50-Failures #5):** emit fields come from the persisted `ClientPurchase` row (`coach_user_id`, `amount_cents`, `currency`, `client_user_id`), never the webhook body. The integration test forges `amount:999999, currency:'xxx'` and asserts they're ignored. Strong.
- **Feature flag default OFF:** `if (process.env.FEATURE_ROMAN_FIRST_PAYMENT !== 'true') return;` — gated, flag-off and flag-explicitly-"false" both tested.
- **`@Optional()` injection + module wiring:** `NotificationsModule` imported into `CheckoutModule`; service exported; handler injection is `@Optional()` so legacy test wiring still constructs. When the flag flips on, the provider resolves (no silent no-op).
- **Ledger row exactly-once + race-safe by constraint:** `coachId @unique` with direct INSERT and P2002→no-op is the correct check-and-set pattern; concurrent first payments / webhook retries race into one winner (50-Failures #28/#29). (The defect is downstream of this, in the notification emit — F1/F2.)
- **No silent swallow:** P2002 is logged structured (`first_payment_already_emitted`); all non-P2002 errors rethrow so the outer tx rolls back and Stripe retries (tested).
- **Migration quality:** additive-only, FK `coachId → User(id)` ON DELETE RESTRICT, unique index, RLS `ENABLE` + `FORCE`, service_role-write + owner-self-SELECT policies (cross-coach read denied). No destructive DDL.
- **PII-minimal payload:** `firstPaymentPayloadSchema` is `.strict()` with only `{amount, currency, clientId}` — no email/name/card/Stripe ids. Body is a fixed string. Deep-link is `tgp://coach/clients/<clientId>`.
- **Webhook framework untouched & sound:** signature verify (HMAC-SHA256, timing-safe, 300s tolerance, rotation), event-id idempotency, event-type allowlist, outer transaction — all pre-existing and not regressed.
- **No banned patterns in new production code; no assistant co-author trailer** (the `Co-authored-by: Bradley Gleave` trailer is the human author — permitted).
- **No new client-hit controller endpoint** → no `@Roles` / `roles-enforced.spec.ts` obligation triggered; no guard bypass introduced.

## Recommended follow-up (all must clear before flag-on / re-merge per R81)

1. **F1/F2 (P0/P1):** Thread the purchase `tx` into the emit so the notification rows are written via `tx.notification.create(...)` (commit-or-rollback with the ledger row). Alternatively, move the emit to a post-commit hook that fires off the committed ledger row only. Add a test that simulates "ledger INSERT succeeds, outer tx rolls back" and asserts **zero** committed notification rows, then re-delivery produces **exactly one**.
2. **F3 (P2):** Correct the PR title and `ROMAN_P4_BACKEND_PLAN.md` to "first payment per coach, forever," OR change to `@@unique([coachId, clientId])` if per-coach×client was the true intent. Decide explicitly.
3. **F4 (P2):** Add a two-distinct-coaches / same-client test asserting both emit.
4. **F5 (P2):** Define and document refund/chargeback behavior; add a test (e.g. refunded first payment either un-records or is explicitly retained-by-design with rationale).
5. **F6 (P2):** Add a Stripe Connect / sub-coach test asserting attribution to `coach_user_id` is intended; document head-coach vs selling-coach semantics.
6. **F7 (P3):** Replace test-only `as unknown as` with typed jest mocks (`jest.mocked` / `Partial<>` + `as` helpers) for consistency.
7. **F8 (P3):** Consider an `AuditService` entry for the once-ever first-payment emit.
