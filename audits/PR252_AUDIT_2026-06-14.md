**Files:** `src/components/onboarding/StripeConnectCard.tsx`, `src/components/onboarding/PermanenceMarker.tsx`

1. `StepTransitionView` — **wired** to `OnboardingLayout`, `LeanQ2`, `LeanQ3`, `LeanQ5`, `OnboardingResults`. ✅
2. `StripeConnectCard` — **not wired** to any production screen. ❌
3. `PermanenceMarker` — **not wired** to any production screen. ❌

Both `StripeConnectCard` and `PermanenceMarker` are fully functional, tested, and flag-safe (they accept an `enabled` prop that defaults to `true` — the caller supplies `featureFlags.romanOnboardingPolish`). The components themselves are correct; the missing piece is that no coach onboarding screen imports or mounts them. The spec calls for `StripeConnectCard` on the Stripe Connect onboarding step and `PermanenceMarker` on the package/pricing input rows; `CoachConnectScreen` (the closest candidate) makes no reference to either component.

This is P2 (not P1) because the flag is OFF and the components are dead code — there is no user-facing regression. However, the "permanence markers" and "card flip" cannot be tested E2E, cannot be validated on device, and cannot be rolled out until the host wiring is added. It is possible the host screens are on a separate (not-yet-merged) branch — if so, this finding documents the gap for the rollout PR to close.

**MMKV persistence audit — `StripeConnectCard`:** The component itself is correctly stateless (the host is responsible for owning `connected: boolean` and persisting it). PR #242's P1 pattern (a once-only gate written but never read) does NOT apply here because: (a) there is no host yet, so there is no gate to audit; and (b) the Stripe Connect "connected" state is derived from a backend query (`ConnectStatusResponse` in `CoachConnectScreen`), not from a local MMKV write. The risk of a write-only gate replicating the PR #242 defect is deferred to whichever PR wires the host — that PR should be audited specifically for this pattern.

**Recommended fix:** Wire `StripeConnectCard` into the Stripe Connect onboarding step and `PermanenceMarker` into the package/pricing input rows in a follow-up PR before the `romanOnboardingPolish` flag is flipped. The follow-up PR must be audited to confirm: (a) `StripeConnectCard.connected` is backed by a server-authoritative source (not client-invented state), and (b) if any locally-persisted "has connected" flag is added to suppress re-showing the flip, it is both written AND read-on-mount (the PR #242 P1 pattern to avoid).

### F2 (P2) — Connect CTA touch target borderline below Android 48dp minimum

**File:** `src/components/onboarding/StripeConnectCard.tsx:282-285`

```ts
cta: {
  marginTop: 8,
  paddingVertical: 14,
  paddingHorizontal: 28,
  borderRadius: 2,
  alignItems: 'center',
},
ctaLabel: {
  fontFamily: 'Inter_500Medium',
  fontSize: 16,
  fontWeight: '500',
},
```

The "Connect Stripe" `HapticPressable` has no explicit `minHeight`. Its rendered height is `paddingVertical * 2 + text height`. With `fontSize: 16` and no `lineHeight` set, React Native's default line height is platform-dependent (typically 1.2–1.25× the font size, so ~19–20px). Total: `14 + 19 + 14 = 47dp` (nominal) to `14 + 20 + 14 = 48dp` at the upper bound. This just meets the iOS 44dp minimum but is at or below the Android 48dp minimum with no `hitSlop` compensation, making the target unreliable across device line-height rounding.

The "Connect Stripe" button is the primary CTA of the front face — the only interactive element on the placeholder card. An undersized target on the most important affordance of this surface is an accessibility gap. Severity P2: no correctness defect; users can tap it and it works; but the target violates Material Design minimum on Android and will fail an accessibility audit. The flag is OFF so this cannot affect production today.

**Recommended fix:** Add `minHeight: 48` to the `cta` style:
```ts
cta: {
  minHeight: 48,
  marginTop: 8,
  paddingVertical: 14,
  paddingHorizontal: 28,
  borderRadius: 2,
  alignItems: 'center',
  justifyContent: 'center',
},
```
This guarantees the Android 48dp minimum regardless of line-height variation and still satisfies iOS HIG.
