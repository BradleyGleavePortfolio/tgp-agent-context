## Per-finding detail

### F1 (P2) ‚Äî `VoiceNotePlayer` play/pause control below minimum touch target

**File:** `src/components/community/VoiceNotePlayer.tsx:190-193`

```ts
control: {
  width: 36,
  height: 36,
  borderRadius: radius.pill,
  alignItems: 'center',
  justifyContent: 'center',
},
```

The `HapticPressable` element that wraps the play/pause `Ionicons` is sized at exactly 36 √ó 36 dp. No `hitSlop` is passed at the call site and `HapticPressable` does not add a default `hitSlop` internally (confirmed by reading `src/components/HapticPressable.tsx` ‚Äî the component forwards `PressableProps` unchanged, no `hitSlop` default). The parent row has `paddingVertical: spacing.sm` (8 dp each side), but React Native's touch target for a `Pressable` is bounded by its own layout dimensions, not the parent's padding. Touch target: **36 √ó 36 dp** ‚Äî fails iOS minimum (44 √ó 44) and Android minimum (48 √ó 48) by 8‚Äì12 dp in each axis.

Severity P2 (not P1): no correctness defect; users can tap the control and it works; but under-sized targets are an accessibility contract gap that hurts users with motor impairments, violates both Apple HIG and Material Design guidelines, and must be fixed before any flag-on rollout. The composer's primary `Send` and `Re-record` controls correctly use `minHeight: 48`; the player button was not matched to that standard.

**Recommended fix:** Increase the control to `minWidth: 44, minHeight: 44` (matching iOS) or add `hitSlop={{ top: 6, bottom: 6, left: 6, right: 6 }}` at minimum. The most natural solution is to bring it to 44 √ó 44 to match the iOS standard: `width: 44, height: 44`, adjusting `size` on the `Ionicons` from 18 to 20 if needed for optical balance.

### F2 (P2) ‚Äî `CommunityVoiceComposerScreen` ships with no dedicated test

**File:** `src/screens/community/__tests__/` (no `CommunityVoiceComposerScreen.test.tsx` exists)

The screen is 118 lines and integrates `useCommunityMe`, `useNavigation`, `useRoute`, `featureFlags`, and `VoiceNoteComposer`. The behaviors not covered by any existing test:

1. **Prereq loading:** `useCommunityMe` in pending/loading state ‚Üí screen renders without passing a null `workspaceId` crash into `VoiceNoteComposer`.
2. **Defense-in-depth flag-off guard:** when the screen is somehow reached with `featureFlags.communityVoiceNotes === false`, it renders the "Voice notes are not available right now." neutral state rather than a blank or crashed screen. The navigator-level flag-off pin test (static source read) does NOT exercise this React render path.
3. **Audience target derivation:** the `useMemo` that converts `route.params` into a `VoiceAudienceTarget` has three branches (hall/cohort/dm); cohort and DM branches carry optional name fields with null-fallbacks that are untested.
4. **`onPublished` ‚Üí `navigation.goBack()`:** the publish callback is wired in the screen, not the composer; untested means a future refactor could silently break the pop.

This is analogous to P2-1 on PR #248 (`CommunityLessonDetailScreen` had no test ‚Äî precisely where P1-1 was missed). While no equivalent P1 exists here (backend contract is clean), the same structural risk applies.

**Recommended fix:** Add `CommunityVoiceComposerScreen.test.tsx` covering: (a) loaded prereq ‚Üí composer rendered with correct `workspaceId`, (b) `useCommunityMe` pending ‚Üí no crash, (c) flag-off guard render, (d) each `target` variant (hall/cohort/dm) producing the correct `VoiceAudienceTarget`, (e) `onPublished` callback fires `navigation.goBack`.

### F3 (P3) ‚Äî No mobile-side telemetry on the voice publish/playback surface

**Files:** all PR screens/components/hooks (no `track(...)` / `AnalyticsEvents` / PostHog references)

The backend PR #397 registered three telemetry events (`voice.upload_issued`, `voice.note_published`, `voice.publish_failed`) and confirmed all are emitted ‚Äî specifically calling out that it "does NOT repeat the PR #396 dead-telemetry pattern." The mobile PR adds no instrumentation at all: no event fired when the user opens the composer, completes a recording, attempts a publish, or plays back a note. A voice recording is an engagement signal that product analytics relies on to understand adoption and failure rates ahead of broader rollout.

This is P3 (polish/hygiene) rather than P2 because: (a) the backend events cover the server-side execution path for publish success/failure, (b) no `FEATURE_COMMUNITY_TELEMETRY` gate is violated (there are no events to misfire), and (c) the v3-2 classroom sibling (PR #248) had the same gap (P2-3 there). Consistent with the repo telemetry pattern being mobile-deferred.

**Recommended fix:** Before flag-on, wire mobile-side events (composer opened, recording started, recording completed, publish attempted, publish succeeded, publish failed, playback started) behind `FEATURE_COMMUNITY_TELEMETRY`. Align event names with the backend `COMMUNITY_TELEMETRY_EVENTS` registry.
