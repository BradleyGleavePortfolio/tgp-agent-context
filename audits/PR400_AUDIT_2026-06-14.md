## Per-finding detail

### F1 (P2) — `GET /coach/home/daily-rings` has no explicit `@Throttle`

**File:** `src/coach/home/coach-home.controller.ts:40–44`

```ts
@Get('daily-rings')
@ApiOperation({ summary: "Today's three-arc completion counts for this coach" })
async dailyRings(@Request() req: AuthedRequest): Promise<DailyRingsResponse> {
  return this.coachHome.getDailyRings(req.user.id);
}
```

No `@Throttle(...)` decorator is present. Per `src/throttler/throttler.config.ts`, the global `default` bucket applies (300 req/min/authed user). The audit brief mandates explicit `@Throttle` on "signing-cost-ish" GET endpoints that run aggregations, and this surface runs up to four Prisma reads in parallel (`countCheckIns` × 2 + `briefOpened` + `countReview` × 2 = four `await`-ed count/findMany calls) every 30 seconds per coach once the cache expires.

The canonical `GET /coach/command-center/:path` (one aggregation-heavy coach read) carries an explicit `@Throttle({ default: { ttl: 60_000, limit: 10 } })` overriding the global default to 10/min. The `coach-home` widget polls on focus with no client-side debounce described; a coach rapidly switching apps could fire many calls within a minute before the 30s cache offers cover.

The route is coach-only (gated by `CoachGuard`), which limits the abuse surface, making this P2 (must-fix before flag-flip) rather than P1.

**Recommended fix:** Add an explicit `@Throttle({ default: { ttl: 60_000, limit: 60 } })` on the `dailyRings` handler (or register a named `COACH_HOME` bucket in `throttler.config.ts`). 60/min is generous for a focus-poll widget while still protecting the aggregation cost:

```ts
@Get('daily-rings')
@Throttle({ default: { ttl: 60_000, limit: 60 } })
@ApiOperation({ summary: "Today's three-arc completion counts for this coach" })
async dailyRings(@Request() req: AuthedRequest): Promise<DailyRingsResponse> {
  return this.coachHome.getDailyRings(req.user.id);
}
```

### F2 (P2) — `DailyRingsResponse` has no Zod `.strict()` response envelope

**File:** `src/coach/home/coach-home.service.ts:49–54`

```ts
export interface DailyRingsResponse {
  checkIns: { reviewed: number; submitted: number };
  brief:    { opened: boolean };
  review:   { reviewed: number; totalConversations: number };
}
```

The response is typed by a plain TypeScript interface. At runtime, `getDailyRings` constructs the response directly from Prisma `.count()` return values and a `length` computation — all numbers/booleans, so there is no immediate data-leak risk. However, the 50-failures #8 doctrine requires runtime response-shape enforcement: any future arc added to the service (e.g. a fourth ring, or `checkIns.pending`) automatically widens the API surface without a deliberate contract decision.

The comparable PR #397 voice notes response shapes (`VoiceNoteViewSchema`, `VoiceUploadTargetSchema`) use `z.strict()` with explicit field enumeration. The `CoachBriefResponse` (sibling coach module) similarly uses a strongly-typed `class` with explicit fields and no extra properties.

**Recommended fix:** Introduce a Zod response schema and parse before returning:

```ts
import { z } from 'zod';

export const DailyRingsSchema = z.object({
  checkIns: z.object({ reviewed: z.number().int().nonnegative(), submitted: z.number().int().nonnegative() }),
  brief:    z.object({ opened: z.boolean() }),
  review:   z.object({ reviewed: z.number().int().nonnegative(), totalConversations: z.number().int().nonnegative() }),
}).strict();

export type DailyRingsResponse = z.infer<typeof DailyRingsSchema>;
```

Then `return DailyRingsSchema.parse(value)` before the cache set and at `zeroedDailyRings()`. This validates that `Prisma.count()` is returning a number (it always does, but the schema becomes the living contract) and immediately surfaces any future arc addition as a schema error rather than a silent API expansion.

### F3 (P2) — `ConversationReview` count query lacks composite `(coach_id, coach_reviewed_at)` index

**File:** `src/coach/home/coach-home.service.ts:190–196`

```ts
this.prisma.conversationReview.count({
  where: { coach_id: coachId, coach_reviewed_at: window },
})
```

**Schema (pre-existing, out of scope for a schema change — surfaced by this PR):**

```prisma
model ConversationReview {
  coach_id          String
  coach_reviewed_at DateTime @default(now())
  …
  @@index([coach_id])        // ← only coach_id
  @@index([client_id])
}
```

The query filters on `coach_id = $1 AND coach_reviewed_at >= $2 AND coach_reviewed_at <= $3`. The existing `@@index([coach_id])` allows PostgreSQL to seek to the coach's rows, but must then scan every `ConversationReview` row for that coach to evaluate the `coach_reviewed_at` window. For a busy coach with hundreds of client threads (all accumulated `ConversationReview` rows), this is a full-partition scan on the index leaf pages. The 30s cache mitigates per-request latency under steady-state polling, but the cold-cache path (first call per day per coach) or after a restart will hit the scan every time.

This index gap is **pre-existing infrastructure** (the `ConversationReview` table and its indexes were established in PR #398 / migration `20261218000000_add_coach_reviewed_at`) and not introduced by this PR. However, this PR is the first consumer of a `coach_reviewed_at`-range filter on `ConversationReview`, making the gap active rather than latent. Per R81 doctrine, the gap is classified as P2 for this PR because the new query shape makes it a real correctness-adjacent performance issue for flag-on.

**Recommended fix:** Add a migration (additive, no destructive DDL) adding the composite index:

```sql
CREATE INDEX IF NOT EXISTS "ConversationReview_coach_id_coach_reviewed_at_idx"
  ON "ConversationReview"("coach_id", "coach_reviewed_at");
```

And add the corresponding `@@index([coach_id, coach_reviewed_at])` to `ConversationReview` in `prisma/schema.prisma`. The existing `@@index([coach_id])` becomes redundant for this query shape once the composite exists (PostgreSQL uses the leftmost prefix), but can be retained for other queries.

### F4 (P3) — In-memory cache `Map` never pruned; unbounded growth on long-uptime servers

**File:** `src/coach/home/coach-home.service.ts:77–120`

```ts
private readonly cache = new Map<string, CacheEntry>();
// …
this.cache.set(cacheKey, { value, expiresAt: now.getTime() + DAILY_RINGS_CACHE_TTL_MS });
```

The cache is keyed by `${coachId}:${utcDay}`. When a coach's entry expires (30s TTL) on the same day, the next call overwrites the same key — no growth. However, entries from prior UTC days are never deleted: the day-part of the key changes at midnight, so `coach-x:2026-06-13` from yesterday stays in the Map forever alongside `coach-x:2026-06-14`. With a roster of N coaches all active at some point in time, the Map grows by N entries per UTC day the server runs without a restart.

On a typical deployment with weekly restarts and a 50-coach roster, this is ~350 stale entries — negligible. On a long-uptime deployment (months) with a large roster, it becomes a real memory pressure source. The issue is strictly P3 (no correctness impact, no security risk) but is the exact pattern 50-failures #42 describes as over-engineering avoidance.

**Recommended fix:** Add a lazy-pruning pass on cache reads — remove entries whose `expiresAt` is more than one day old:

```ts
private pruneCache(now: Date): void {
  const cutoff = now.getTime() - 24 * 60 * 60 * 1_000; // entries older than 1 day
  for (const [key, entry] of this.cache.entries()) {
    if (entry.expiresAt < cutoff) this.cache.delete(key);
  }
}
```

Call `this.pruneCache(now)` at the top of `getDailyRings` before the cache lookup. Alternatively, bound the Map to a fixed maximum size (e.g. 200 entries with LRU eviction), or use `@nestjs/cache-manager` with a Redis backend to avoid in-process state entirely.

### F5 (P3) — No PostHog / analytics telemetry for the ED.2 daily-rings surface

**Files:** `src/coach/home/coach-home.service.ts`, `src/coach/home/coach-home.controller.ts`

Neither `getDailyRings` nor any of the three arc methods emit a PostHog event or call `AnalyticsService`. The `AnalyticsService` (no-op when `POSTHOG_KEY` unset, never throws) is injected by `CoachOnboardingService` and `CoachController` in the adjacent `src/coach/` module, so the pattern is established and the module is already in scope.

For a widget that is polled on every Coach Home focus, basic telemetry (`coach.daily_rings_fetched`, carrying ring counts as properties) would give direct signal on: (a) how often coaches open the home screen (widget usage rate), (b) average completion state across rings, and (c) whether flag-on at the mobile layer is driving actual backend calls. The PR #397 comparison point registered and emitted three events for a new surface; the ED.2 surface is at minimum as product-signal-relevant.

**Recommended fix:** Inject `AnalyticsService` into `CoachHomeService` (it is a globally-provided service) and emit on a cache miss (flag-on path only, so the polling nature of the widget doesn't flood PostHog):

```ts
// emit only on a real Prisma read (cache miss), not on every call
this.analytics.capture(coachId, 'coach.daily_rings_fetched', {
  check_ins_submitted: value.checkIns.submitted,
  check_ins_reviewed:  value.checkIns.reviewed,
  brief_opened:        value.brief.opened,
  review_reviewed:     value.review.reviewed,
  review_total:        value.review.totalConversations,
});
```

No PII; all numeric/boolean properties.

## Hectacorn bar

The architecture is sound: a read-only aggregation service that composes four existing repositories without writing, touching no new Prisma models, scoped entirely by the calling coach's JWT identity, behind a default-OFF per-request flag, with a correctly-invalidating in-memory cache that avoids database round-trips on every focus-poll. The `DailyRingsRepo` narrow interface pattern is a deliberate, well-motivated R80 choice. Stripe or Linear would ship this data model.

What holds it back from the full hectacorn bar: the GET endpoint runs four database reads per cold-cache call and has no explicit rate-limit bucket of its own (F1); the response shape has no Zod `.strict()` guard against silent future schema drift (F2); the `ConversationReview` count query hits a suboptimal index path that will become visible at scale (F3); and there is no telemetry to know whether the widget is being used at all — making flag-flip readiness hard to assess from the PostHog dashboard (F5). Resolving F1–F3 and optionally F5 closes the gap to a fully shippable production surface.
