## Per-finding detail

### F1 (P2) — `POST reviewed` write route has no explicit `@Throttle`

**File:** `src/check-ins/coach-check-ins.controller.ts`

```ts
@Post('clients/:client_id/check-ins/:check_in_id/reviewed')
@HttpCode(200)
async markReviewed(
  @Request() req: AuthedRequest,
  @Param('check_in_id') checkInId: string,
) {
  return this.checkIns.markReviewedByCoach(req.user.id, checkInId);
}
```

No `@Throttle(...)` decorator is present. The route is a write path that executes two Prisma calls per request (a `findFirst` authorization check + an `update`). The global default bucket (300 req/min/user per `throttler.config.ts`) is the only rate-limiting control. The audit brief and the `throttler.config.ts` comment table both identify write routes as requiring an explicit named bucket. The adjacent `markRead` on `CoachMessagingController` (`POST clients/:id/messages/read`) also lacks an explicit `@Throttle`, but that is pre-existing and out of scope here; this finding is scoped to the new line introduced by this PR.

The surface is coach-only (gated by `CoachGuard`) so client-side abuse is blocked, and coaches have limited roster sizes, making this P2 (must-fix before flag-flip) rather than P1. The canonical brief classifies missing throttle on write routes as P2.

**Recommended fix:** Add `@Throttle({ default: { ttl: 60_000, limit: 60 } })` (or a named `check-ins-review` bucket in `throttler.config.ts`) on the `markReviewed` handler. 60/min/user is generous for a coach-acknowledgement action; adjust to match product expectations.

### F2 (P2) — `markReviewedByCoach` returns unbounded raw Prisma row

**File:** `src/check-ins/check-ins.service.ts:363–375`

```ts
async markReviewedByCoach(coachId: string, checkInId: string) {
  await this.assertCheckInOfCoach(coachId, checkInId);
  const flagOn = isCoachReviewedAtEnabled();
  const updated = await this.prisma.checkIn.update({
    where: { id: checkInId },
    data: {
      reviewed_by_coach: true,
      ...(flagOn ? { coach_reviewed_at: new Date() } : {}),
    },
  });
  return updated;  // ← full Prisma CheckIn row, all columns
}
```

Prisma `update` without a `select` returns the entire model row. The `CheckIn` model has 14+ columns including `user_id`, `coach_id`, `notes`, `weight_kg`, `sleep_hours`, and the newly added `coach_reviewed_at`. The controller passes `return updated` directly to the HTTP response. Any future column added to `CheckIn` (e.g. a medical PII field) will automatically appear in this endpoint's response without an explicit contract decision.

The 50-Failures #8 doctrine requires response shapes to be explicitly declared (Zod `.strict()` or a narrowed `select` clause). Comparable service methods (`coachReviewForClient`, `stampConversationReview`) use explicit `select` or return typed inline objects. The `markReadByCoach` sibling returns `{ updated: result.count }` — a narrow, intentional shape.

**Recommended fix:** Either (a) add `select: { id: true, reviewed_by_coach: true, coach_reviewed_at: true }` to the `prisma.checkIn.update` call and return that narrow shape, or (b) post-process `updated` through a Zod `.strict()` schema, e.g.:

```ts
const AckSchema = z.object({
  id: z.string(),
  reviewed_by_coach: z.boolean(),
  coach_reviewed_at: z.string().datetime().nullable(),
}).strict();
return AckSchema.parse({
  id: updated.id,
  reviewed_by_coach: updated.reviewed_by_coach,
  coach_reviewed_at: updated.coach_reviewed_at?.toISOString() ?? null,
});
```

### F3 (P2) — Two-step find-then-update does not re-assert `coach_id` in the UPDATE WHERE clause

**File:** `src/check-ins/check-ins.service.ts:363–385`

```ts
async markReviewedByCoach(coachId: string, checkInId: string) {
  await this.assertCheckInOfCoach(coachId, checkInId);  // findFirst { id, coach_id }
  // … flag check …
  const updated = await this.prisma.checkIn.update({
    where: { id: checkInId },         // ← only id; coach_id not re-asserted
    data: { reviewed_by_coach: true, … },
  });
  return updated;
}

private async assertCheckInOfCoach(coachId: string, checkInId: string) {
  const row = await this.prisma.checkIn.findFirst({
    where: { id: checkInId, coach_id: coachId },
    select: { id: true },
  });
  if (!row) throw new NotFoundException('Check-in not found');
  return row;
}
```

The authorization check confirms `coach_id = coachId` in a separate `findFirst`, but the `update` scopes only by `id`. Between the two calls, if `coach_id` were mutated (e.g., via a reassignment operation), the update would proceed on a row the authorizing coach no longer owns. `CheckIn.coach_id` is currently set at upsert time and no existing service path mutates it, making this latent. However, the correct pattern eliminates the two-round-trip TOCTOU window and is a single atomic write:

```ts
async markReviewedByCoach(coachId: string, checkInId: string) {
  const flagOn = isCoachReviewedAtEnabled();
  try {
    const updated = await this.prisma.checkIn.update({
      where: { id: checkInId, coach_id: coachId },   // ← scoped atomically
      data: {
        reviewed_by_coach: true,
        ...(flagOn ? { coach_reviewed_at: new Date() } : {}),
      },
      select: { id: true, reviewed_by_coach: true, coach_reviewed_at: true },
    });
    return updated;
  } catch (err: unknown) {
    if ((err as any)?.code === 'P2025') throw new NotFoundException('Check-in not found');
    throw err;
  }
}
```

This collapses two round-trips into one, re-asserts ownership in the same database operation, and eliminates the TOCTOU window entirely.

**Note:** Prisma `update` throws `P2025` when no record matches the `where` clause, which correctly covers both the "not found" and "wrong coach" cases with the same 404 response (no existence leakage). The 50-Failures #28 (race-safe check-and-set) rubric is technically satisfied by idempotent `now()` semantics on `coach_reviewed_at`, but the read-then-write pattern for the authorization step is still a latent risk worth closing.

**Recommended fix:** Consolidate into a single `update({ where: { id: checkInId, coach_id: coachId }, ... })` with P2025 catch → 404, as shown above. This simultaneously resolves F2 if the `select` clause is included.

### F4 (P3) — No test for sub-coach attribution on `ConversationReview` marker

**File:** `test/messaging.service.spec.ts`

`markReadByCoach` uses `threadCoachId = client.coach_id ?? coachId` when stamping the marker. For a sub-coach caller, `client.coach_id` is the head coach's id, so the marker is written under the head coach's id rather than the sub-coach's id. No test asserts this attribution. The analogous gap was a P2 finding in PR #395 (F6); this is P3 here because sub-coach delegation is not the primary audience for the ED.6 pill and the code path is technically covered by the existing `requireClientCoachId` → `assertClientOfCoach` plumbing.

**Recommended fix:** Add a test case where `coachId` is a sub-coach's id, `client.coach_id` is the head coach's id, and assert that the resulting `conversationReview` row has `coach_id` equal to the head coach's id.

### F5 (P3) — No PostHog / analytics telemetry for the coach-reviewed surface

**Files:** `src/check-ins/check-ins.service.ts`, `src/messaging/messaging.service.ts`

Neither `markReviewedByCoach` nor `stampConversationReview` / `coachReviewForClient` emits any PostHog event or analytics call. PR #397 (voice notes) registered and emitted three events (`voice.upload_issued`, `voice.note_published`, `voice.publish_failed`) for a comparable new surface; that PR explicitly passed the "registered-and-emitted" telemetry check. The ED.6 surface — which is the first backend signal visible to clients about coach engagement — has no equivalent observability.

Specific actionable events that would be worth capturing: `coach.check_in_reviewed` (on `markReviewedByCoach` when flag ON), `coach.thread_review_stamped` (on `stampConversationReview`), and `client.competence_pill_read` (on `coachReviewForClient` when the marker exists). Properties should carry ids and timestamps only, no PII.

**Recommended fix:** Register the above event names in the appropriate event registry and emit them at the relevant call sites. If the product decision is explicitly "no telemetry for this surface," document that decision in the service file.
