## Findings

| ID | Sev | Area | Finding |
|----|-----|------|---------|
| F1 | **P2** | Storage-key correctness / audit traceability | In `service.create`, media storage keys are built with `buildMediaSeeds(workspaceId, randomUUID(), input.media)` — a **throwaway** `randomUUID()` is passed as the `postId` path segment, then discarded. The real post id is generated separately by Prisma `@default(uuid())` inside `createPostWithMedia`. So the create-time key `community-classroom/{ws}/{randomUUID}/{kind}/{randomUUID}` embeds a `postId` segment that matches **no** real post. `attachMedia` correctly uses the real `post.id`, so the two write paths produce inconsistent key shapes. No security impact (key is still workspace-prefixed + trailing `randomUUID` → unguessable, tenant-bound, no IDOR), but it breaks the code's own stated invariant ("intrinsically scoped to this workspace + lesson") and makes storage keys un-groupable / un-auditable by post id. The service spec only asserts the `community-classroom/{ws}/` prefix, so it does not catch the mismatch. |
| F2 | **P2** | Unbounded write / work amplification | `ClassroomMediaInputDto[]` on `CreateClassroomPostDto.media` and `AttachClassroomMediaDto.media` has `@IsArray` + `@ValidateNested` but **no `@ArrayMaxSize`**. `@ArrayMaxSize` is the established repo convention (`src/lessons/lessons.dto.ts`, `meal-plans`, `workout-builder`, etc.). A coach can submit thousands of media entries in one request → a large `createMany` + a sequential per-asset signed-URL loop (`buildUploadTargets` calls `createSignedUploadUrl` once per asset) inside one transaction. Bounded to coaches (auth'd, throttled at `COMMUNITY_POSTS_PER_MIN`), so not a P0/P1, but it is an unbounded-input write the rest of the codebase caps. |
| F3 | **P2** | Dead telemetry / commit-message overstatement | The 3 new `community.classroom.*` event-name constants are added to `community-events.ts` and pinned in `posthog-event-names.spec.ts`, but **no classroom code path emits them** — there is no `capture()`/`track()`/`COMMUNITY_TELEMETRY_EVENTS` reference anywhere under `src/community/classroom/`. The commit message ("classroom telemetry events … Emitted only when FEATURE_COMMUNITY_TELEMETRY === 'true'") implies live emission on publish/schedule/media-issue that does not exist. The §9 pin is intentionally exhaustive and the repo explicitly tolerates name-before-emitter (PR #370 note), so this is logged P2 (not P0 dead code): the names are a contract placeholder, but `lesson_published` / `lesson_scheduled` / `media_upload_issued` will silently never fire until an emitter is wired. |
| F4 | **P2** | Missing throttle on reads | The two read routes (`listFeed`, `getOne`) carry no `@Throttle`, unlike all 5 write routes (`COMMUNITY_POSTS_PER_MIN`). The student feed mints a signed download URL **per media asset** (`mediaView` → `createSignedDownloadUrl`), so an unthrottled hot loop on the feed is a signing-cost / storage-API amplification vector. A global throttler may apply, but the slice does not set an explicit read bucket the way sibling community surfaces do. |
| F5 | P3 | Test hygiene (R0-permitted) | Two `@ts-expect-error` directives in test files (`community-classroom.service.spec.ts:158`, `pinned-ordering.spec.ts:65`) for structural partial mocks. Each carries a one-line justification, which R0 explicitly permits in tests; the production source is 100% clean of `@ts-ignore`/`@ts-expect-error`/`as any`/`as never`/`as unknown as`. Flagged P3 only for R81 completeness. |
| F6 | P3 | Re-publish ordering edge | `publish` stamps `published_at = post.published_at ?? now` (stable across re-publishes — correct), but a coach who first publishes immediately, then edits `release_at` to a future time and re-publishes, transitions a `published` lesson back to `scheduled` while keeping the original `published_at`. Visibility stays correct (the predicate keys off `release_at`, not status), but the feed ordering anchor (`published_at DESC`) for a re-scheduled lesson reflects the original publish, not the new schedule. Cosmetic ordering only; no visibility or auth impact. Undocumented; worth a one-line comment or test. |

## Evidence

### F1 (P2) — create-time storage key embeds a throwaway postId
`community-classroom.service.ts:346-363`:
```ts
const seeds =
  input.media && input.media.length > 0
    ? this.buildMediaSeeds(workspaceId, randomUUID(), input.media)  // <-- throwaway postId
    : [];
const created = await this.repo.createPostWithMedia({ ... media: seeds });
```
`buildMediaSeeds` (lines 238-265) builds `community-classroom/${workspaceId}/${postId}/${m.kind}/${randomUUID()}`. The `postId` here is the discarded `randomUUID()`, never the persisted `created.id`. Contrast `attachMedia` (line 462) which correctly passes `post.id`:
```ts
const seeds = this.buildMediaSeeds(post.workspace_id, post.id, media);
```
Repository generates the real id via Prisma default (`schema.prisma`: `id String @id @default(uuid()) @db.Uuid`). Net: create-path keys are workspace-scoped and unguessable (safe) but their `{postId}` segment is meaningless. Spec gap: `community-classroom.service.spec.ts:231` asserts only `expect(seeds[0].storageKey).toContain(\`community-classroom/${WS_A}/\`)`.

### F2 (P2) — no array-size cap on media input
`community-classroom.dto.ts:166-170` and `207-211`:
```ts
@IsOptional() @IsArray() @ValidateNested({ each: true }) @Type(() => ClassroomMediaInputDto)
media?: ClassroomMediaInputDto[];           // CreateClassroomPostDto — no @ArrayMaxSize
...
@IsArray() @ValidateNested({ each: true }) @Type(() => ClassroomMediaInputDto)
media!: ClassroomMediaInputDto[];           // AttachClassroomMediaDto — no @ArrayMaxSize
```
`buildUploadTargets` (service:286-308) loops `createSignedUploadUrl` once per asset; `createPostWithMedia` / `attachMedia` do a single `createMany` of arbitrary size inside `$transaction`.

### F3 (P2) — telemetry names registered, never emitted
`community-events.ts` adds `classroomLessonPublished` / `classroomLessonScheduled` / `classroomMediaUploadIssued`. Grep across `src/community/classroom/**` for `capture|track|posthog|COMMUNITY_TELEMETRY_EVENTS|classroomLesson|classroomMedia` returns **no emit site**. `publish` and the media write paths perform no telemetry call.

### F4 (P2) — reads unthrottled
`community-classroom.controller.ts`: all 5 write handlers carry `@Throttle({ default: { ttl: 60_000, limit: COMMUNITY_POSTS_PER_MIN } })`; `listFeed` (147-157) and `getOne` (159-167) carry none. Each returned post triggers `Promise.all(media.map(mediaView))` → one signed-URL mint per asset.

## What's correct (and why it earns the AUDITED verdict)

**R0 doctrine — production src clean.** Scoped sweep over the 9 changed `src/` files for `Coming soon`, `@ts-ignore`, `.catch(()=>undefined)`, `as unknown as`, `as any`, `as never` → **zero hits**. (Repo-wide pre-existing hits in unrelated files — `account-deletion.service.ts`, `admin/ptm`, `admin/reports` — are out of this PR's scope.)

**Commit trailers clean.** `git log b19fee89~1..b19fee89 --format=%B | grep -iE 'co-author|claude|opus|sonnet|assistant|generated-by|gemini|gpt|copilot'` → only `Co-authored-by: Bradley Gleave <bradley@bradleytgpcoaching.com>`, a permitted legitimate human co-author. No assistant-attribution trailer (would be P0).

**Migration — additive, reversible, RLS-FORCE.** Wrapped in `BEGIN/COMMIT`; `CREATE TABLE` only (no `ALTER` on existing tables); `coach_id` FK `ON DELETE RESTRICT` (a coach with lessons can't be silently deleted), media `post_id` FK `ON DELETE CASCADE`. Both tables `ENABLE` **and** `FORCE ROW LEVEL SECURITY`. Coach policy `FOR ALL` with both `USING` and `WITH CHECK` keyed on `app.is_community_workspace_coach(workspace_id)`. Member `SELECT` policy enforces `status='published' AND (release_at IS NULL OR release_at <= now()) AND soft_deleted_at IS NULL AND (workspace-member OR shares-cohort)`. Media member-`SELECT` re-derives the same predicate through a `post` join (`EXISTS (... p.status='published' AND ...)`), so a non-member cannot resolve a `storage_key` for signing. Documented rollback path. Indexes cover the feed `(workspace_id, cohort_id, status, release_at)` and pinned `(workspace_id, cohort_id, pinned, pinned_order)` orderings.

**Release lock — clock-skew-safe, unbypassable.** Visibility is decided by the pure `isStudentVisible` predicate (`release.feature.ts`), which keys off `release_at` vs an injected `now`, **not** the `status` column, so a `scheduled`/`published` mismatch can never leak a locked lesson. The same predicate is mirrored in three places that the design explicitly keeps in sync: the service `readablePost`, the repository `listForStudent` filter (pushed INTO the query so the page, its cursor anchor, and the overflow `nextCursor` row all share the predicate — no release-locked row can become a public cursor token), and the RLS member-select policy. "Unlock" is purely (a) the timestamp passing or (b) an authorized coach re-publishing — there is no student-reachable unlock endpoint and no payload that flips visibility.

**Release-lock race (procedure step 7).** Two workspace coaches concurrently editing `release_at` is last-write-wins on a Prisma `update`, but both are authorized owners of the same workspace by design — no privilege gained, no student-visible window opened (the predicate re-evaluates `release_at` at read time). Not a security race; at most a cosmetic ordering edge (see F6).

**Authorization.** Every write handler carries `@UseGuards(JwtAuthGuard, RolesGuard, CommunityFeatureFlagGuard, CommunityClassroomEnabledGuard)` + `@Roles('coach','owner')`; reads carry the master guard + `@Roles('student','coach','owner')` (students must reach reads). Coach ownership is enforced in the **service** (`assertCoach` / `coachOwnedPost` / `isCoach` via `CommunityAccessService.isWorkspaceCoach`, with platform `owner` allowed). Cross-tenant reads → **404** (never 403, existence never leaks) via `readablePost`; a member who can read but is not the coach → **403** on write paths via `coachOwnedPost`. All `:postId` / `:workspaceId` params use `ParseUUIDPipe({ version: '4' })`. The new controller is auto-covered by `roles-enforced.spec.ts` (reflection-based discovery of every `AppModule` controller — `CommunityModule` → `CommunityClassroomModule` is registered, and every handler carries `@Roles`), so the procedure's "new routes in roles-enforced.spec.ts" is satisfied implicitly with no manual allowlist entry.

**Feature flag default OFF.** `resolveClassroomFlag()` returns `process.env.FEATURE_COMMUNITY_CLASSROOM_POSTS === 'true'` — any other value (incl. unset) is OFF. Guard read per-request (no boot-cache → runtime kill without redeploy). Gates all 5 write routes; reads intentionally ride only the master flag so already-released lessons survive a mid-rollout kill. The e2e reflection suite asserts every mutating route carries the kill-switch and every read route does not, and that the flag-off path throws the typed 503 before any handler runs.

**Media / signed URLs — anti-IDOR.** Object access is always a time-limited signed URL minted at read time (`createSignedDownloadUrl`) or upload time (`createSignedUploadUrl`); never a public/permanent URL. Per-kind size caps (`CLASSROOM_MEDIA_MAX_BYTES`: video 500MB / audio 100MB / pdf 50MB / image 25MB) are enforced server-side **before** any URL is minted. Keys embed the workspace prefix + a `randomUUID()` (unguessable, tenant-bound — no cross-workspace replay, no path traversal). Signed-URL TTL is clamped to `[60s, 24h]`. Storage-not-configured degrades to `url=null` (disabled tile) on reads and a clean 400 on writes — never a 500. Per-asset signing failures are isolated so one bad key never blanks a feed.

**DTO strictness (R8).** Request DTOs are class-validator; the global pipe (`main.ts:117`) sets `whitelist: true, forbidNonWhitelisted: true, transform: true`, so unknown request properties are rejected. All response schemas are Zod `.strict()` (`ClassroomMediaSchema`, `ClassroomPostSchema`, `ClassroomUploadTargetSchema`, and both response wrappers), parsed on the way out.

**N+1 / unbounded (procedure step 9).** The feed joins media via a single Prisma `include` (no per-post loop). `runFeedQuery` fetches `take: limit + 1` to derive `nextCursor`; `clampLimit` caps at `[1, 50]` defensively even if the DTO is bypassed. Stale/foreign cursor degrades to the first page rather than throwing.

**Transactions.** `createPostWithMedia` and `attachMedia` wrap the post+media writes in `$transaction` so a partial write can never orphan a post or asset. Soft-delete sets `soft_deleted_at` + `status='archived'` (row retained for audit; reads exclude it).

**Cron / background jobs (procedure step 10).** None added. Scheduled release is purely lazy/time-evaluated at read (`release_at <= now()`) — there is no background job to be non-idempotent or unlocked.

**Tests (procedure step 11).** 45 DB-free unit specs plus an RLS suite. Failure-path assertions present: cross-tenant/non-member → 404 (`service.spec.ts:192,293`), non-coach write → 403 (`:182`), release-lock 404 to student (`:273`), coach sees `release_locked=true` (`:284`), soft-deleted → 404 (`:304`), oversize media rejected before signing (`:205`, asserts `createSignedUploadUrl` not called), workspace-bound key (`:230`), storage-unconfigured degrades to `url=null`/400 not 500 (`:314`, `:240`), configurable TTL (`:325`). RLS spec statically asserts `FORCE`, coach `FOR ALL` (USING+WITH CHECK), and member `SELECT` (published+released+not-deleted+membership) on both tables, with a live cross-tenant/release-lock/soft-delete/cohort suite gated on a disposable Postgres. The e2e reflection suite pins guard topology + flag-off. The §9 PostHog pin updated 6 → 9 events.
