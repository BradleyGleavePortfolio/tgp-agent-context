## Findings

### P1-1 — Lesson-detail response schema mismatch breaks the happy path (functional, flag-gated)

**File:** `src/api/communityClassroomApi.ts:109-111`, `:221-225`
**Backend:** `/tmp/fix-pr396/src/community/classroom/community-classroom.service.ts:521-528`; DTO `community-classroom.dto.ts` (`ClassroomPostResponseSchema`).

Mobile parses the detail (`GET /community/classroom/:postId`) response with:

```ts
const ClassroomPostDetailResponseSchema = z
  .object({ post: ClassroomPostSchema })
  .strict();
```

The backend `getOne` returns and validates `{ post, upload_targets: [] }`:

```ts
async getOne(user, postId): Promise<ClassroomPostResponse> {
  const post = await this.readablePost(user, postId, now);
  return ClassroomPostResponseSchema.parse({
    post: await this.postView(post, now),
    upload_targets: [],   // "Empty on edits/reads" per the DTO
  });
}
```

Because the mobile schema is `.strict()`, the unexpected `upload_targets` key makes the Zod parse **throw**, which `call()` wraps as a `contract` error. Result: **every successful lesson-detail fetch fails validation and renders the "We could not load this lesson. Please try again." error state** instead of the lesson. The feed endpoint is unaffected — both sides agree on `{ posts, next_cursor }`.

Severity P1 (not P0): no production impact today (flag OFF in prod, route unregistered), but it is a guaranteed break on first enablement against the live backend, not an edge case. Fix: either drop `.strict()` on the detail schema, accept `upload_targets` optionally (`upload_targets: z.array(...).optional()`), or align the mobile detail read to the backend `ClassroomPostResponse` shape and select `.post`.

### P2-1 — No test for `CommunityLessonDetailScreen`

**File:** `src/screens/community/__tests__/` (absent: `CommunityLessonDetailScreen.test.tsx`)

The PR adds a 331-line detail screen but ships **zero tests** for it. The untested behaviors are precisely the ones the checklist flags as failure paths: `getLesson` error → calm not-found (cross-tenant/release-locked 404), `media.url === null` → disabled "preparing" tile, `release_locked` → media fully suppressed + "unlocks later" note, flag-off defense-in-depth guard, and the title a11y announcement. Had a detail-screen test existed it would very likely have surfaced **P1-1** (a real fetch against a mock mirroring the backend `{ post, upload_targets }` shape would have failed the parse). The feed screen, card, hook, and flag-off cases are otherwise well covered.

### P2-2 — No entry point and no deep-link registration for the classroom routes

**Files:** `src/navigation/CommunityNavigator.tsx`; no caller navigates to `CommunityClassroom`.

With the flag on, the routes register but are unreachable from any tab, card, or deep link — the feed is wired internally (`navigate('CommunityLessonDetail', …)`) but nothing opens the feed itself. This is acceptable for flag-off scaffolding, but the surface is non-functional until a future lane adds the entry point; it should be tracked so the rollout PR does not assume the surface is reachable.

### P2-3 — No telemetry on a consumption surface

No instrumentation is added (lesson opened, feed paged, media-tile state). The checklist's item 11 (gate telemetry on `FEATURE_COMMUNITY_TELEMETRY`) is therefore vacuously satisfied, but a classroom is a content-engagement surface where view/open analytics are typically expected. Add behind the telemetry gate before rollout.
