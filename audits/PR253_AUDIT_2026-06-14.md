## Per-finding detail

### F1 (P1) — `inverse addExercise` / `inverse removeExercise` corrupt D-045 delete-tracking state

**File:** `src/screens/coach/CoachWorkoutBuilderScreen.tsx:1317–1355`

The `applyInverse` executor in the screen has two cases whose interaction with the autosave D-045 delete-tracking refs is incorrect:

**Case 1 — `inverse removeExercise` (undo of `addExercise`)**

When the undo of a user `addExercise` fires, this case (line 1317) correctly removes the row from `rows` _and_ mirrors the D-045 delete-tracking that `removeRow` does:

```ts
case 'removeExercise': {
  setRows((cur) => {
    const idx = cur.findIndex((r) => r.clientId === op.clientId);
    if (idx === -1) return cur;
    const removed = cur[idx] as DraftExerciseRow;
    // Mirror removeRow's delete-tracking (D-045).
    deletedKeysRef.current.add(removed.clientId);
    const sig = rowCompositeSignature(removed);
    const list = deletedSignaturesRef.current.get(sig) ?? [];
    list.push(removed.clientId);
    deletedSignaturesRef.current.set(sig, list);
    return cur.filter((_, i) => i !== idx);
  });
  return;
}
```

This is intentional and correct in the autosave-only scenario: if the autosave insert for the forward `addExercise` was in-flight, the server will confirm the row and the adoption refetch will carry it back — the D-045 guard needs to suppress it. This case is fine.

**Case 2 — `inverse addExercise` (undo of `removeExercise`) — THE BUG**

When the undo of a user `removeExercise` fires, this case re-inserts the row with a fresh `clientId` but does **not** clear the D-045 tracking left by the original `removeRow`:

```ts
case 'addExercise': {
  setRows((cur) => {
    const next = cur.slice();
    const restored: DraftExerciseRow = {
      clientId: generateClientId(),  // Fresh clientId — old one remains in deletedKeysRef
      row_id: undefined,
      exercise_external_id: op.row.exerciseExternalId,
      // ...
    };
    const at = Math.min(Math.max(op.atIndex, 0), next.length);
    next.splice(at, 0, restored);
    return next;
  });
  return;
}
```

The original `removeRow` added the row's old `clientId` to `deletedKeysRef` and its composite signature to `deletedSignaturesRef`. Those entries are never cleared by this undo path.

**What breaks:** When the autosave sends the `remove_exercise` to the server and the subsequent refetch/adoption cycle runs, the adoption effect's D-045 guard at lines 944–964 fires:

- **Path 1 (server row_id known):** `rowIdToClientIdRef` maps the row's `row_id` → old `clientId`. The guard sees the old `clientId` in `deletedKeysRef` and drops the row from `keptExercises`. `setRows` replaces the full local state with the `keptExercises` list — which no longer contains the re-added exercise. The undo was visually successful for a few seconds, then silently reverted by the refetch.

- **Path 2 (row was id-less at removal time):** The composite signature in `deletedSignaturesRef` matches the re-added row's server entry when it comes back, and it is dropped for the same reason.

The D-045 cleanup at lines 988–995 only prunes `deletedKeysRef` when the server row_id is **absent** from live server truth (i.e. the delete successfully landed). After a remove → undo, the server still holds the row, so `liveServerRowIds.has(rowId)` is `true` — the cleanup condition is `!liveServerRowIds.has(rowId)`, so the old entry is never pruned. The stale delete-tracking persists for the entire session, and every adoption cycle after the undo re-drops the row.

This is a silent data corruption on the feature's most obvious use-case: accidentally removing an exercise and immediately undoing it. The `commandStack.size` correctly reflects the undo, the toast fires, but the on-screen change is reverted by the next autosave refetch (typically within 300–500ms of the autosave debounce). The integration test suite does not cover `removeExercise → undo`, so this defect is undetected.

**Recommended fix:**

In the `inverse addExercise` case, before re-inserting the row, clear the original row's D-045 entries using the snapshot's `rowId` and the row's composite signature:

```ts
case 'addExercise': {
  // Clear D-045 delete-tracking for the row being restored, so the
  // autosave adoption cycle does not re-drop it.
  const snapshotClientId = op.row.rowId
    ? rowIdToClientIdRef.current.get(op.row.rowId)
    : undefined;
  if (snapshotClientId) {
    deletedKeysRef.current.delete(snapshotClientId);
  }
  // Also clear signature-based tracking (covers id-less rows).
  const sig = rowCompositeSignature({
    clientId: snapshotClientId ?? '',
    row_id: op.row.rowId,
    exercise_external_id: op.row.exerciseExternalId,
    display_name: op.row.displayName,
    sets: op.row.sets,
    reps_or_duration_seconds: op.row.repsOrDurationSeconds,
    rest_seconds: op.row.restSeconds,
    weight_lbs: op.row.weightLbs,
    superset_group_id: op.row.supersetGroupId,
    notes: op.row.notes,
  });
  const pending = deletedSignaturesRef.current.get(sig);
  if (pending && pending.length > 0) {
    pending.shift(); // consume one entry (FIFO, matching removeRow's push)
    if (pending.length === 0) deletedSignaturesRef.current.delete(sig);
  }
  setRows((cur) => { /* ...existing re-insert logic... */ });
  return;
}
```

Add a regression integration test: user removes an exercise (with autosave on), then undoes — assert the row is still present after the post-autosave adoption refetch fires.

> Mitigating context, not exoneration: `EXPO_PUBLIC_FF_MWB_UNDO` defaults `false` unconditionally, so this cannot reach production until ops flips it. The defect is latent, which keeps it at P1 rather than P0. R81 audits the merged surface in any regard.

### F2 (P2) — Undo toast has no `accessibilityLiveRegion`

**File:** `src/screens/coach/CoachWorkoutBuilderScreen.tsx:1484–1491`

The undo toast renders as:

```tsx
{undoEnabled && undoToast ? (
  <View style={styles.undoToast} testID="mwb-undo-toast">
    <Text style={[typography.caption, { color: sc.textPrimary }]}>
      {undoToast}
    </Text>
  </View>
) : null}
```

There is no `accessibilityLiveRegion` on the `<View>` (or the inner `<Text>`), and no `AccessibilityInfo.announceForAccessibility` call on toast appearance. VoiceOver (iOS) and TalkBack (Android) users who trigger undo via the gesture or button receive no audible confirmation that the revert completed — the toast is purely visual. The task brief explicitly lists "live-region announce on undo" as a CRITICAL mobile requirement for this feature.

The `UndoButton` itself has a correct `accessibilityLabel="Undo last change"` and `accessibilityState={{ disabled: !canUndo }}`, so the button's state is correctly communicated. The gap is the confirmation feedback.

**Recommended fix:**

```tsx
{undoEnabled && undoToast ? (
  <View
    style={styles.undoToast}
    testID="mwb-undo-toast"
    accessibilityLiveRegion="polite"
    accessibilityRole="status"
  >
    <Text style={[typography.caption, { color: sc.textPrimary }]}>
      {undoToast}
    </Text>
  </View>
) : null}
```

`accessibilityLiveRegion="polite"` causes the toast content to be announced by screen readers when it appears without interrupting in-progress speech. Add a test assertion on `accessibilityLiveRegion` to the toast show case in `coachWorkoutBuilderAutosave.test.tsx`.

### F5 (P3) — `applyInverse` `useCallback` empty dependency array communicates a wrong contract

**File:** `src/screens/coach/CoachWorkoutBuilderScreen.tsx:1315`

```ts
const applyInverse = useCallback((op: InverseOp): void => {
  switch (op.kind) {
    // ... reads deletedKeysRef.current, deletedSignaturesRef.current,
    //     rowCompositeSignature, setRows, setName, setType, generateClientId
  }
}, []);  // ← empty array
```

All of the referenced values are either React-stable (`setRows`/`setName`/`setType` dispatchers are stable across renders; refs are accessed via `.current`; `rowCompositeSignature` and `generateClientId` are module-level functions), so the hook produces a correct and non-stale closure today. The command-stack hook defends against a stale closure by routing calls through `applyInverseRef.current` (updated on every render), so even if `applyInverse` were re-minted it would be picked up correctly.

However, the empty array communicates "no dependencies" when the function body clearly depends on several values. A maintainer adding a new branch that reads a non-stable local variable (e.g. `planId`, `undoEnabled`) will not be warned by the linter to add it to the dep array. The pattern is fragile-by-contract.

**Recommended fix:** Either add the genuinely used deps (`setRows, setName, setType, deletedKeysRef, deletedSignaturesRef, rowCompositeSignature`) with an `eslint-disable-next-line react-hooks/exhaustive-deps` + comment explaining why they are all stable, OR extract the non-React-state parts into module-level helpers so the callback's dep list is genuinely empty.

### F6 (P3) — Undo toast always says "Twenty steps still in the bank" regardless of stack size

**File:** `src/screens/coach/CoachWorkoutBuilderScreen.tsx:1419`

```ts
showUndoToast(romanBuilderUndoToast.success({ depth: commandCapacity }));
//                                                     ↑ capacity = 20 always
```

`commandCapacity` is `commandStack.capacity`, which is the configured depth (default 20). It does not reflect how many undos remain. The `romanBuilderUndoToast.success` signature takes a `depth` arg; the copy reads "Reverted. Twenty steps still in the bank." This is always true (capacity is 20) but never informative — after 19 undos the toast still says "Twenty steps still in the bank."

If the intent was to communicate "the buffer lets you go back twenty steps" (absolute capacity reassurance), the copy should be "Up to twenty steps available." If the intent was "you have N more undos left," the arg should be `commandStack.size` (the live count **after** the pop). As shipped, a coach on their 20th undo sees the same toast as on their first, giving no signal that the history is exhausted.

**Recommended fix:** Either (a) pass `commandStack.size` (post-pop) and update the copy to "Reverted. {N} steps remaining." / "Reverted. Last step." for `size === 0`, or (b) change the copy to "Reverted." (no count) and remove the `depth` arg as unnecessary complexity.

## What's correctly implemented (do not regress)

- **Bounded stack:** depth 20, FIFO eviction — no memory leak on long sessions. The `stackRef` + `useState(size)` split correctly avoids double-invoke bugs in StrictMode.
- **`inverseOf` pure function:** total over all 5 forward ops, exhaustiveness guard (`_never`) ensures a compile error when a new action kind is added without an inverse. Directly unit-tested.
- **No-pop-on-failure:** `undo()` optimistically pops from the ref before calling `applyInverseRef.current`, then restores the action on rejection — the coach can retry. Re-throw propagates the error to the `onUndo` handler which surfaces the Roman error toast via the existing `romanGenericError` stem.
- **In-memory only (correct by design):** the stack is session-only (component state, wiped on unmount). The task brief's "state persistence" concern is correctly addressed — the spec explicitly says fearless-experimentation tool, not durable history.
- **Race-safe double-undo:** the ref (`stackRef`) is mutated synchronously before `await applyInverseRef.current(op)`, so two rapid undo calls cannot pop the same entry. Verified by the sequential-undo test.
- **Flag default OFF unconditionally:** `mwbUndo: readFlag('EXPO_PUBLIC_FF_MWB_UNDO', false)` — `false` is the hardcoded fallback, not `isDev`. Flag is read per-render via `featureFlags.mwbUndo` (the module-level object is evaluated at import time; tests mock it with live getters to allow per-test toggling). Flag-off renders zero UI residue: no `UndoButton`, no toast, no Pan gesture constructed (`UndoButton` never mounts). Statically verified by `mwbUndoFlagOff.test.tsx`.
- **Push ref pattern:** `pushUndoActionRef` is null when `undoEnabled` is false — zero push overhead in flag-off builds. Populated via a `useEffect` that cleans up on flag-flip or unmount.
- **Server reconciliation architecture:** undo issues no direct server calls. All inverse ops flow through the same `setRows`/`setName`/`setType` state mutations that forward ops use, so the existing autosave diff picks up the change and sends the reconciling PATCH/PUT through the same idempotent channel. No new endpoints, no new network surface. Correct.
- **`applyInverse` executor ref pattern in hook:** `applyInverseRef.current = applyInverse` on every render keeps the latest closure without making `undo` referentially unstable.
- **Theme tokens:** `bgSurface`, `accentText`, `textMuted`, `border` all confirmed present in `src/theme/tokens.ts`. No raw hex in any PR file. No `semanticColors.surface` (non-existent token) used.
- **Touch target:** `minWidth: 44, minHeight: 44` + `hitSlop={8}` on `UndoButton` Pressable — meets iOS 44dp and Android 48dp targets.
- **Quiet-luxury doctrine:** no TODO/FIXME/XXX in `src/screens` or `src/components` (confirmed). No `fontWeight: '700'|'800'`. No "Coming soon". `UndoButton` docstring quiet-luxury-correct (no emoji, no exclamation, line icon `arrow-undo-outline`). Roman copy file uses no contractions, no hype, no emoji.
- **RNTL v14:** every `render()` and `renderHook()` call is `await`ed across all four new test files.
- **Reanimated thread safety:** the `Gesture.Pan().runOnJS(true)` pattern is explicitly chosen because the `onUndo` handler touches React Query / autosave state (JS thread). No Reanimated worklet-crossing state writes. No `useNativeDriver: false`. No Animated API used at all — gesture is RNGH only, transition is `opacity` CSS-style (no animation); `useReduceMotion` is N/A (no motion surface in this PR).
- **`applyInverse` → `inverse removeExercise` (undo-of-addExercise):** correctly mirrors D-045 delete-tracking to suppress the server confirmation of a just-added row the coach immediately undid (see F1 for the converse failure).
