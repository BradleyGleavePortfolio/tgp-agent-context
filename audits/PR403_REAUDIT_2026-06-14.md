## 4. New Findings

### F6 (P2) — F4 throttle decorators are not regression-pinned

**Finding.** PR #403 adds the required throttle decorators on the regime/refund write handlers, but it does not add a regime/refund controller metadata test or smoke test that would fail if those decorators are later removed.

**Evidence.** The only regime/refund throttle hits are the controller source lines themselves: refund decision at `src/regimes/refund-decisions.controller.ts:50`, regime promote at `src/regimes/regimes.controller.ts:65`, regime update at `src/regimes/regimes.controller.ts:82`, and regime archive at `src/regimes/regimes.controller.ts:94` (`PR403_throttle_test_search.txt:1-10`). The `diff throttle tests` section is empty, meaning this PR did not add a throttle pinning test (`PR403_throttle_test_search.txt:162-162`). The repository already uses throttle metadata tests as the established regression pattern, including auth named-throttler metadata, messages block/unblock/report metadata, wearables exact throttle metadata, and checkout payment-intent throttle metadata (`PR403_throttle_test_search.txt:28-54`, `PR403_throttle_test_search.txt:112-123`, `PR403_throttle_test_search.txt:136-142`, `PR403_throttle_test_search.txt:144-153`).

**Impact.** R79 says every code-level fix must have a test pinning the new behavior; the prompt explicitly listed “F4 = throttle decorator presence test or smoke test.” Without a test, the fix is not squeaky-clean under R81 even though the decorators are present today.

**Required fix.** Add a regime/refund controller contract spec that reads `THROTTLER:LIMITdefault` and `THROTTLER:TTLdefault` from `RefundDecisionsController.prototype.decide`, `RegimesController.prototype.promote`, `RegimesController.prototype.update`, and `RegimesController.prototype.archive`, asserting `10/60000` for refund decisions and `30/60000` for regime writes.
