## P0 findings
- [src/screens/client/CheckoutReturnScreen.tsx:221-238, tgp-agent-context/design/MOBILE_APP_DESIGN_INTELLIGENCE_2026-05-30.txt:336-337, tgp-agent-context/rules/R0_DECACORN_QUALITY.md:113-124] R0 blocks the success confirmation as an **Empty Confirmation**. The successful paid state is a static icon, title (`You are subscribed`), one body line, and one button; there is no dedicated success animation, haptic sequence, progressive reveal, or premium closure moment on the checkout peak/end state. The mandatory R0 mobile playbook says a major purchase/completion followed by static confirmation is an anti-pattern, and the R0 rule says any listed anti-pattern must be flagged as P0. Fix by investing in the checkout return as the peak moment: fire a success haptic when `isPaying` becomes true, animate the confirmation/check state with reduced-motion fallback, make the copy emotionally specific to the purchased package, and provide a clear next-step hook (e.g. unpack deliverables / message coach) without adding more than one primary decision.

## R0 (Decacorn) review
- **Peak-End Rule:** Fails the R0 bar. The checkout success state is the peak moment of a paid buyer journey, but the shipped state is static and emotionally under-invested [src/screens/client/CheckoutReturnScreen.tsx:221-238]. Apple-quality treatment would make the payment confirmation feel safe, premium, and complete with a calibrated success haptic, a short success animation, copy tied to the package name, and a next-step that answers “what happens now?” without adding decision load.
- **One decision per high-load screen:** Mostly passes. `PackageDetailSurface` has one primary buyer action (`Continue to payment` / `Pay ...`) and secondary trust copy is non-actionable [src/screens/client/packageDetail/PackageDetailSurface.tsx:145-173]. Checkout error states present retry/cancel, which is acceptable for recovery, but should be visually ordered as one recommended path plus an escape hatch [src/screens/client/BrandedCheckoutWebViewScreen.tsx:406-426].
- **Miller's Law / ≤5 actionable elements:** Buyer checkout passes with a close control and one pay CTA visible around the critical purchase decision [src/screens/client/PackageCheckoutScreen.tsx:229-262]. Coach edit is denser but the action cluster is below the critical buyer purchase moment; it has save, preview, share/manage/subscribers/archive/content actions in a long scroll, which is acceptable for a coach power-user editor but should eventually move destructive/archive behind a more explicit secondary menu [src/screens/coach/payments/CoachPackageEditScreen.tsx:471-583].
- **CALM framework on error states:** Functionally acceptable but not decacorn-level. The webview errors are clear and actionable, distinguish blocked-origin, expired session, Stripe outage, and network failure, and include retry/cancel actions [src/screens/client/BrandedCheckoutWebViewScreen.tsx:270-303, src/screens/client/BrandedCheckoutWebViewScreen.tsx:308-326, src/screens/client/BrandedCheckoutWebViewScreen.tsx:389-426]. They lack Light Feedback (haptics) and any calming motion/brand reassurance, so they pass correctness but not peak polish.
- **Forbidden anti-patterns:** Empty Confirmation is present and is filed as P0 above [src/screens/client/CheckoutReturnScreen.tsx:221-238]. I did not find permission-front onboarding, feature-dump first screen, unescapable streak architecture, gamification mismatch, or obvious inconsistency-tax in the changed commerce surfaces. Polish-as-afterthought risk is visible in the static checkout success treatment; the success moment needs to be designed as part of the flow, not after the money path works.
- **Apple / Notion / Google lens:** Apple would not ship a paid checkout completion without a proportionate emotional confirmation. Notion would keep the preview surface shared and content-first, which this PR does well. Google would require the return-link contract to be robust across both the webview short-circuit and app-level linking fallback, and would add the missing linking-config test.
