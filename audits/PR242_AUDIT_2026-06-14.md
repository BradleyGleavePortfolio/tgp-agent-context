## Per-finding detail

### F1 (P1) — Persisted once-only gate is write-only; ED.3 re-fires across sessions

ED.3 is specified as "THE moment" that must fire **exactly once per coach, ever** (`firstPaymentGate.ts` header; `FirstPaymentWowHost.tsx` step 4). The durable contract is supposed to be the MMKV gate `roman.ed3.first-payment-seen.${coachId}`. But the gate is **never read**:

- `markFirstPaymentSeen(coachId)` is called on dismiss (`FirstPaymentWowHost.tsx:158`) — the **write**.
- `hasSeenFirstPayment(coachId)` (`firstPaymentGate.ts:31`) is the **read** — and `git grep` shows it is referenced **only in its own test** (`firstPaymentGate.test.ts`). Neither the Host nor `useFirstPaymentNotification` ever calls it.

What actually suppresses a re-show is purely **in-memory and session-scoped**:
- `useNotifications.ts:138` `firedLatchRef` — a per-coach `useRef`, reset to `null` on every fresh hook mount.
- `FirstPaymentWowHost.tsx:99` `dismissedCoachIdsRef` — a `useRef<Set>`, empty on every fresh host mount.

Neither survives an app restart. `fetchNotifications(null)` is a paged GET that returns notifications **regardless of read state** (`src/services/notificationsApi.ts:198`; reading marks `read=true` but does not remove the row from the page). So on the next launch: `firedLatchRef` is fresh `null`, `dismissedCoachIdsRef` is fresh empty, the persisted gate is never consulted → the poll finds the still-present `FIRST_PAYMENT` notification and calls `onFirstPayment` again → the celebration re-overlays. This repeats on every cold start until the notification falls out of the first page.

The Host docstring ("once-only contract … MMKV once-only gate") and the `FirstPaymentWowHost.test.tsx` header ("the gate write must persist BEFORE the UI returns … so a re-render can never re-arm the celebration") both **describe a guarantee that is not wired**. The test only mocks `markFirstPaymentSeen` (write) and never exercises a read-on-mount, so it cannot catch this.

This is the single most important emotional beat in the coach app, and the as-shipped behavior is "celebrate the first payment again every time you reopen the app for ~30 notifications' worth of history." Not a crash, so P1, not P0 — but a real correctness defect on the headline feature, and the kind of thing Apple/Linear/Stripe would never ship (hectacorn bar: fail).

**Recommended fix:** in `FirstPaymentWowHost` (or the hook), gate `onFirstPayment` on `await hasSeenFirstPayment(coachId)` before showing — i.e. seed the in-memory dismissed set from the persisted gate on mount, or have the hook skip-fire when the gate is already closed. Add a regression test that mounts the host with the persisted gate pre-set to `'true'` and asserts the overlay never appears even when the notification is present.

> Mitigating context, not exoneration: the flag `romanFirstPaymentWow` defaults **OFF**, so this cannot reach production until ops flips it. The defect is latent, which is why it is P1 rather than higher. But R81 backfill audits the merged surface "in any regard," and a dark flag does not make a wired-wrong once-only contract correct.

### F2 (P2) — Pan scrubber sets React state per pointer-move (low-end Android frame risk)

`ProgressChartCard.tsx` builds `Gesture.Pan().runOnJS(true)` and calls `emitIndex(...)` in both `onBegin` and `onUpdate`. `emitIndex` calls `setSelectedIndex(index)` — a React state update — on every pointer move while scrubbing, which re-renders the whole card (re-running `useAnimatedProps`, re-deriving `selectedPoint`, re-rendering the SVG tree) on each move. The haptic itself is correctly throttled to crossovers (`lastEmitted.current`), but the `setState` is not. On a low-end Android device this is the classic JS-thread-synced-to-gesture pattern that produces dropped frames during the drag. The draw-in animation is fine (UI-thread Reanimated); the scrubber is the risk. Heuristic-level, not measured, hence P2.

**Recommended fix:** drive the scrubber position through a Reanimated shared value on the UI thread (worklet `onUpdate`), and only bridge to JS state on crossover (when the snapped index changes), mirroring the haptic throttle. Or memoize so a same-index update is a no-op (`if (index === selectedIndex) return;`).

### F3 (P2) — `mutedGold` used as a fill against its documented token constraint

`src/theme/tokens.ts:42` documents `mutedGold` as "ONLY for founding-tier badge typography, never as fill." `ProgressChartCard.tsx` uses `colors.mutedGold` (and `withAlpha(colors.mutedGold, …)`) as the **fill** of the PR star `<Polygon>` and the two glow `<Circle>`s. This violates the token's stated usage rule and, per the contrast table in the tokens header, `mutedGold` fails WCAG AA except as ≥14pt bold label text — so as a small decorative fill it is also a low-contrast surface. P2 (design-system contract + a11y contrast), not P1, because the PR surface is currently behind the OFF flag and PR detection is disabled for the only live consumer (bodyweight). Still ships in the dark path and would surface the moment a lift series opts in.

**Recommended fix:** use an approved accent fill (e.g. `colors.romanAccent` / `colors.forest`) for the star + glow, or get the token constraint formally relaxed for chart-marker fills.

### F4 (P2) — TODO tokens in production source (`src/hooks/useNotifications.ts`)

The field is dead and the intended "stagger each particle by its delay" (per the `useAnimatedStyle` comment) is not actually implemented. Cosmetic; the burst still reads fine. P3.
