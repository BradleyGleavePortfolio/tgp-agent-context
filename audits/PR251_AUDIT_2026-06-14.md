## Per-finding detail

### F1 (P2) — `voice_note_transcript` search result navigates to the wrong route

**File:** `src/screens/community/CommunityFindScreen.tsx:84-86`

```ts
case 'voice_note_transcript':
  navigation.navigate('CommunityThread', { postId: result.targetId });
  break;
```

**Backend reference:** `src/community/search/search-indexer.listener.ts:89` (backend worktree)

The search indexer stores `voice_note_transcript` results with `targetId: note.id` — the `CommunityVoiceNote` record's own UUID, which lives in `community_voice_notes` and is **not** a community post ID. `CommunityThread` expects a community `Post` UUID and fetches via `GET /community/workspaces/:wsId/posts/:postId`. When voice note indexer wiring lands (a future PR opts the voice-note write service into the indexer), a user who taps a transcript search result will pass a voice note UUID to the thread screen, which will receive a 404 from the backend and render the calm "We could not load this thread" error state rather than the voice note content.

Neither classroom_lesson → `CommunityLessonDetail` nor event → `CommunityEventDetail` has this problem: the classroom lesson ID IS the targetId of the post record, and the event ID routes to a dedicated event screen. The voice note is the only kind that lacks a dedicated detail screen accessible by its own ID.

Additionally, `CommunityLessonDetail` and `CommunityEventDetail` are only registered in the navigator when `featureFlags.communityClassroom` and `featureFlags.communityEvents` are `true` respectively. If `communitySearch` is ON but either of those flags is OFF, tapping a `classroom_lesson` or `event` result will attempt to navigate to an unregistered route. React Navigation logs a warning in development and silently no-ops in production, leaving the user on the search screen with no feedback. This is a dependent-flag ordering gap that should be resolved before `communitySearch` flips on.

**Recommended fix:** Add a `voice_note_transcript` case that either (a) opens a dedicated `CommunityVoiceNoteDetail` screen once that route exists, or (b) renders a dismissible "this content is not yet viewable from search" sheet until the voice detail route lands. For `classroom_lesson` and `event` cases, guard the navigation with the respective feature flag: if `!featureFlags.communityClassroom` return early with a benign "not available" alert or no-op; same for `communityEvents`.

### F2 (P2) — Missing flag-off navigator pin tests

**Files:** absent `src/navigation/__tests__/communitySearchFlagOff.test.ts` and `src/navigation/__tests__/coachCommunityWearablePromptsFlagOff.test.ts`

Every prior community lane that adds a flag-gated navigator route ships a corresponding static pin test:
- v3-2 classroom: `communityClassroomFlagOff.test.ts`
- v3-3 voice notes: `communityVoiceFlagOff.test.ts`

These tests pin the contract that (1) the flag ternary appears in source order before the `<Stack.Screen>` registration, (2) the ternary occurs exactly once, and (3) the flag defaults to `false` in the feature-flags file. This PR adds two new flag-gated routes but no such tests. The missing coverage means a future editor could accidentally move the screen registration outside the gate ternary with no failing test.

For `CoachCommunityNavigator`, the missing test is especially important because the wearable-prompts surface is COACH-ONLY health data — a mis-gate could expose it to clients.

**Recommended fix:** Add `communitySearchFlagOff.test.ts` and `coachCommunityWearablePromptsFlagOff.test.ts` following the established pattern in `communityVoiceFlagOff.test.ts`: static source reads of `CommunityNavigator.tsx` and `CoachCommunityNavigator.tsx` that pin the flag ternary precedes the screen name, the registration count is exactly one, and `featureFlags.ts` records `false` as the default.

### F3 (P2) — Both new screens ship without dedicated screen-level tests

**Files:** absent `src/screens/community/__tests__/CommunityFindScreen.test.tsx` and `src/screens/community/__tests__/CommunityWearablePromptsScreen.test.tsx`

`CommunityFindScreen` (306 lines) integrates `useCommunityMe`, `useCommunitySearch`, `dedupeById`, debounce state, `featureFlags`, FlatList, and `open()` navigation routing. `CommunityWearablePromptsScreen` (385 lines) integrates `useCommunityMe`, three mutation hooks, role check (`isCoachOrOwner`), `busyId` per-prompt mutation tracking, and all six state branches (flag-off, prereq-loading, prereq-error, not-coach, list-loading, list-error, empty, populated). This is structurally identical to the PR #248 situation where the untested `CommunityLessonDetailScreen` masked the P1 schema mismatch. While no equivalent P1 exists here (backend contract is clean), the same structural risk applies: F1 (the voice note routing gap) would very likely have been caught by a screen test that exercised the `open()` handler with a `voice_note_transcript` result.

**Recommended fix:**

`CommunityWearablePromptsScreen.test.tsx` should cover: (a) flag-off guard, (b) prereq loading/error, (c) non-coach renders unavailable, (d) missing clientId renders unavailable, (e) populated list renders WearablePromptCards, (f) generate mutation fires with clientId, (g) dismiss/actOn mutation fires with promptId and sets busyId.

### F4 (P3) — `SearchBar` clear button below minimum touch target

**File:** `src/components/community/SearchBar.tsx:49-61`

```ts
<Pressable
  onPress={() => onChangeText('')}
  accessibilityRole="button"
  accessibilityLabel="Clear search"
  hitSlop={8}
  testID="community-search-clear"
>
  <Ionicons name="close-circle" size={18} color={semanticColors.textMuted} />
</Pressable>
```

The `Pressable` wraps a single `Ionicons` of size 18 dp with no explicit `width`/`height`/`minWidth`/`minHeight` style. `hitSlop={8}` extends the touch area by 8 dp on each side, yielding an effective tap region of approximately 34 × 34 dp. The parent `SearchBar` container is itself approximately 34–36 dp tall (fontSize 15 input + paddingVertical 8 × 2), so the Pressable's layout is constrained within that height regardless. Both values fall below the iOS HIG minimum of 44 × 44 dp and the Android Material Design minimum of 48 × 48 dp.

Severity P3 rather than P2: the clear button is supplementary (the user can also backspace); the primary search input and all action buttons elsewhere meet the threshold. The `WearablePromptCard` action buttons correctly use `minHeight: 44`.

**Recommended fix:** Add `minWidth: 44, minHeight: 44` to the `Pressable` style (no `hitSlop` needed once layout is correct), adjusting the icon's `size` from 18 to 20 for optical balance if desired. Alternatively, add `hitSlop={{ top: 13, bottom: 13, left: 13, right: 13 }}` to compensate without a layout change.

## What's correctly implemented (do not regress)

- **Backend contract — full parity (no PR #248 repeat).** `SearchResponseSchema` matches `community-search.dto.ts` field-for-field: `{ version: 1, query, results, nextCursor, tookMs }` with `SearchResultRowSchema` (`id, kind, targetId, cohortId, authorId, excerpt, createdAt`), both `.strict()`. `PromptViewSchema`, `PromptListResponseSchema`, `GenerateResponseSchema`, `PromptSourceViewSchema` all match `wearable-prompts.dto.ts` exactly, all `.strict()`. `dismiss` and `actOn` return a single `PromptView` (not a versioned envelope) — the mobile `call(PromptViewSchema, ...)` wrapper is correct. Zero Zod contract mismatches.
- **Feature flag defaults OFF unconditionally.** `communitySearch: readFlag('EXPO_PUBLIC_FF_COMMUNITY_SEARCH', false)` and `communityWearablePrompts: readFlag('EXPO_PUBLIC_FF_COMMUNITY_WEARABLE_PROMPTS', false)` — both hardcoded `false`, not `isDev`. Routes registered inside ternaries `{featureFlags.communitySearch ? ... : null}` and `{featureFlags.communityWearablePrompts ? ... : null}` in their respective navigators. Both screens carry defense-in-depth runtime guards (neutralUnavailable states) in case they are somehow reached.
- **Search disabled when no workspace or empty term.** `useCommunitySearch` gates on `Boolean(opts.workspaceId) && trimmed.length > 0`; no unbounded "match everything" request can fire. The screen additionally trims the input before passing it to the hook.
- **Debounce at the screen boundary.** `SEARCH_DEBOUNCE_MS = 300` using `setTimeout`/`clearTimeout` in a `useEffect` — the standard React Native pattern for search debouncing. The `SearchBar` is a pure controlled input; the screen owns the debounce lifecycle.
- **Cursor-pagination + bounded reads.** `useInfiniteQuery` with `initialPageParam` + `getNextPageParam: (p) => p.nextCursor ?? undefined`; `SEARCH_PAGE_LIMIT = 20`, `SEARCH_PAGE_LIMIT_MAX = 50`; `onEndReachedThreshold={0.4}`; `AbortSignal.timeout` per read. No unbounded fetch.
- **Cross-tenant safety.** Search RLS is enforced server-side: non-members receive 403 (mapped to `forbidden`); cohort/soft-delete visibility enforced at the repository level. No client-side filtering that could produce cross-tenant leakage — all filtering is via server-supplied cursor and the server's own membership gate.
- **Coach-only wearable surface — multi-layer defense.** (1) Registered in `CoachCommunityNavigator` only. (2) Screen checks `featureFlags.communityWearablePrompts`. (3) Screen checks `useCommunityMe` role: `isCoachOrOwner = role === 'coach' || role === 'owner'`; non-coach/owner sees the neutral unavailable state, never any prompt. (4) Backend enforces `@Roles('coach', 'owner')` on every wearable-prompts endpoint. No raw health VALUE is surfaced (source pills show `metricKey + observedValue` which the coach is entitled to see via the client's consent grant).
- **Generate mutation race handled.** Generate button is `disabled={generate.isPending}` with `accessibilityState={{ disabled: true, busy: true }}` so rapid taps cannot queue multiple inflight generate calls.
- **Quiet-luxury doctrine — clean.** No `TODO`/`FIXME`/`XXX`, no `fontWeight: '700'|'800'` (heaviest is `'600'`), no "Coming soon" string in any PR file across all 19 changed files.
- **Theme tokens only.** Zero raw hex literals in production src. All color references via `semanticColors.{bgPrimary, bgSurface, border, accent, accentText, textPrimary, textMuted, textOnAccent}` — all confirmed present in `src/theme/tokens.ts`. `Colors.textPrimary` / `Colors.surface` / `Colors.background` in `CoachCommunityNavigator` confirmed in the legacy `Colors` export in `src/theme/index.ts`.
- **No animations in any PR file.** `useReduceMotion`, `Reanimated`, and `useNativeDriver` are absent from all 19 changed files. N/A check passes.
- **RNTL v14 await pattern.** Every `renderHook(...)` call in the three test files is `await`-ed. Compliant.
- **TanStack Query v5 pattern.** `useInfiniteQuery` with `initialPageParam`; `useMutation` without deprecated options; `mutateAsync` in test assertions; `await waitFor(...)` for post-mutation state reads. Compliant.
- **No AsyncStorage.** None of the 19 PR files import or reference AsyncStorage.
- **Accessibility.** `CommunityFindScreen` FlatList: `accessibilityRole="list"` + count label; rows wrapped in `role="listitem"`. `SearchResultRow` Pressable: `accessibilityRole="button"` with `"${meta.label}. ${result.excerpt}"` folded label. `SearchEmptyState`: `accessibilityRole="text"` with folded `"${title}. ${body}"` label. `CommunityWearablePromptsScreen` FlatList: `accessibilityRole="list"` + count label. `WearablePromptCard` action buttons: `accessibilityRole="button"`, `accessibilityState={{ disabled: busy }}`, `minHeight: 44`. `WearablePromptSourcePill`: `accessibilityRole="text"` with `"Source: <label> <value>"` label. `AccessibilityInfo.announceForAccessibility` fires once per unique `term:count` pair when search results land. Loading states carry `accessibilityState={{ busy: true }}` + `accessibilityRole="progressbar"` + `accessibilityLabel` on `ActivityIndicator`. The only gap is F4 (clear button).
- **dedupeById used at both list boundaries.** Search results and wearable prompts both pass through `dedupeById(...)` before rendering, preventing duplicate keys on cursor overlap.
- **Zod boundary on every read.** Every network response passes through `schema.parse(res.data)` before entering React state; a drifted shape throws a typed `contract` CommunityApiError, never feeding malformed data into the UI.
