The notification rows now ride the ambient purchase `tx` end to end.
- `notifications.service.ts:297-331` — `createNotification(input, tx?)` resolves `const db = tx ?? this.prisma`, threads `tx` into `getPreferences(input.user_id, tx)`, and writes via `db.notification.create(...)` (was `this.prisma.notification.create`).
- `emitters/first-payment.emitter.ts:65-110` — `emit(coachId, payload, tx?)` passes `tx` into BOTH `createNotification` calls (inapp + push).
- `coach-first-payment.service.ts:96-160` — `tryEmitFirstPayment(tx, input)` does `tx.coachFirstPaymentNotification.create(...)` then `this.emitter.emit(coachId, {...}, tx)`.
- `checkout-webhook-handler.service.ts:548,912` — BOTH callsites pass the outer `tx` (and now `event.id` as `correlationId`); `maybeEmitFirstPayment` returns early `if (!this.coachFirstPaymentService || !tx)` so the emit only runs inside a transaction. `WebhookTx = Prisma.TransactionClient` (line 26) — type-compatible to the service signature.

NEW integration test verified to ASSERT (not just call):
`__tests__/first-payment-tx-rollback.integration.spec.ts` drives the REAL `CoachFirstPaymentService → FirstPaymentEmitter → NotificationsService` against a FakeDb that models tx-buffered vs autocommit writes:
```

These assertions would fail under the old (autocommit) code → genuine regression lock.
