## P1 findings
- [src/ai/ai.service.ts:79-82, src/ai/ai.service.ts:335-342, src/ai/ai.service.ts:555-564, src/ai/ai.service.ts:608-615] The 12,000 daily cap still is not a hard bound on TOTAL provider tokens because the up-front reservation depends on a character-count heuristic (`ceil(chars / 4 * 1.15) + 600`) rather than a guaranteed tokenizer upper bound, and the `actual > reserved` reconcile path increments `tokens_used` after the provider call without any cap guard. DTOs cap history/message characters, but they do not cap provider tokens; token-dense inputs can materially exceed the chars/4 estimate. Under concurrent requests, each call can pass the atomic reservation using the low estimate, spend more real input tokens than reserved, and then push the ledger above `DAILY_TOKEN_QUOTA` only after the provider cost has already been incurred. This means prior P1(a) is only partially fixed: provider totals are persisted for ordinary successful responses, but the reserve amount still does not safely bound total-token spend. Fix: reserve a true upper bound before the call (provider tokenizer count, or a conservative worst-case token-per-character bound including system/context/history/user text plus max output), or enforce input limits that make the estimate a proven upper bound; do not allow the post-call `actual > reserved` charge path to be the first place the cap learns about real spend.

## P2 findings
- [src/ai/ai.service.ts:397-405, src/ai/ai.service.ts:439-448, src/ai/ai.service.ts:473-477] If a provider response contains no usable text but does include usage, the code never records that usage: it sets `actualTokens` only inside the non-empty-text branch, marks the response as deterministic fallback, and the `finally` block refunds the full reservation. That contradicts the “reconcile against provider FULL total” claim for a provider call that did run and reported usage. Fix: read and preserve provider usage as soon as the response is available, even when the content is empty and the user-facing answer falls back; refund only when the provider did not run or no billable usage is available by policy.

## Verification of prior findings and fixer claims
- P1(a), total-token accounting: FALSE / partially fixed. Successful Perplexity responses with content now use `response.usage.total_tokens`, and Anthropic content responses use `tokensIn + tokensOut`, so the persisted ledger can reflect provider FULL totals on those paths. However, the up-front reservation is a heuristic estimate rather than a hard upper bound, and `actual > reserved` is reconciled after the spend with an unguarded increment, so concurrent calls can still exceed the 12,000 daily cap when real tokenization beats the estimate.
- P1(b), refund on failure: verified for provider failure/fallback paths. The provider section is wrapped in `try/finally`; when the provider throws and the service falls back, `modelUsed === 'fallback'` with `actualTokens === null`, so `reconcileDailyTokens(..., actual=0)` refunds the entire token reservation. The refund is an atomic `updateMany` decrement guarded by `tokens_used >= refund`, so it cannot underflow below zero; there is only one `finally`, so there is no obvious double-refund path. A successful provider response with text and no usage is not wrongly refunded because `modelUsed` remains the provider and `actualTokens` stays null, so the conservative reservation is kept.
- P2, day-key rollover: verified fixed. `reserveDailyTokens()` captures the UTC quota date and returns it; `chat()` passes that exact `quotaDate` into every reconcile/refund call, so reserve and reconcile/refund target the same row across midnight.
- Race / atomicity: partially verified. The reserve path still uses a DB-atomic guarded `updateMany` with `tokens_used <= DAILY_TOKEN_QUOTA - cost`, so concurrent requests cannot overshoot the reserved estimate. This does not fully protect the hard 12,000 total-token cap because the estimate can under-reserve and the later `actual > reserved` increment is unguarded.
- Scope/schema: schema unchanged. Source changes are under `src/ai/`, but the overall diff also touches three test files outside `src/ai/`.
