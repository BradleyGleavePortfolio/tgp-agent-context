## Executive summary
PR #493 is not mergeable and is not close. It declares the D-H6-1 “13-column” audit_log contract but ships a different 13 columns, different role/action/entity naming, different request_id type, no user_agent column, and tests that pin the wrong contract as if it were locked. CI is red for R75, R74, R82 proof, and build-and-test; the logs identify concrete failures, not noise. The service API also violates the H6A/H6C contract by owning the transaction instead of accepting the caller transaction, which means the downstream H6C canonical wrap cannot compose with it. Verdict is FINDINGS with multiple BLOCKERs.

## Findings

### F1 — audit_log schema is not the locked D-H6-1 schema
- **Severity:** BLOCKER
- **Doctrine cite:** D-H6-1; R82; R125; H6A builder brief §“What this slice ships” lines 42–55
- **File + line:** `prisma/migrations/20261226000000_create_audit_log/migration.sql:45-58`; `prisma/schema.prisma:6737-6754`; `test/audit-log/audit-log.migration.spec.ts:34-48`
- **Observation:** The brief requires columns `actor_user_id`, `actor_role`, `entity_type`, `entity_id`, `request_id uuid`, `ip_inet`, and `user_agent`; the migration instead creates `tenant_id`, `actor_id`, `actor_type`, `resource_type`, `resource_id`, `request_id text`, and `ip_address`, with no `user_agent`. This is not a harmless naming choice; it changes the contract H6B/H6C are supposed to compose against.
- **Required fix:** Replace the migration, Prisma model, service types, and tests with the exact locked D-H6-1 column names/types, or obtain and document a new operator decision explicitly superseding D-H6-1 before this PR proceeds.
- **Evidence:** H6A brief lists the locked table as `actor_user_id`, `actor_role`, `entity_type`, `entity_id`, `request_id uuid`, `ip_inet`, and `user_agent`; the migration creates `tenant_id`, `actor_id`, `actor_type`, `resource_type`, `resource_id`, `request_id text`, `ip_address`, and no `user_agent` at `migration.sql:45-58`. The Prisma model mirrors the wrong shape at `schema.prisma:6737-6754`, and the migration spec encodes the wrong column list at `audit-log.migration.spec.ts:34-48`.

### F2 — withAuditLog API violates the H6A/H6C same-transaction contract
- **Severity:** BLOCKER
- **Doctrine cite:** D-H6-5; H6A builder brief lines 82–86; H6C canonical wrap pattern lines 51–73
- **File + line:** `src/audit-log/audit-log.service.ts:47-65`; `src/audit-log/audit-log.types.ts:52-62`; `test/audit-log/audit-log.service.spec.ts:392-445`
- **Observation:** The brief requires `withAuditLog<T>(tx: Prisma.TransactionClient, args: AuditLogArgs, op: () => Promise<T>)`, using the caller-provided transaction. The implementation exposes `withAuditLog(ctx, fn)` and internally starts `this.prisma.$transaction`, so a caller already inside `prisma.$transaction(async (tx) => ...)` cannot pass that `tx` through. That breaks the H6C canonical pattern before H6C even starts.
- **Required fix:** Change the public API to accept the caller transaction client and audit args separately, remove the internal transaction ownership from `withAuditLog`, update tests to assert caller-provided `tx` is used, and update docs/types accordingly.
- **Evidence:** The service starts its own transaction at `audit-log.service.ts:47-65`; the H6C brief’s canonical pattern calls `this.auditLog.withAuditLog(tx, {...}, () => tx.user.update(...))`, which this implementation cannot satisfy.

### F3 — GDPR erasure path performs UPDATE against a table designed and tested as no-UPDATE
- **Severity:** BLOCKER
- **Doctrine cite:** D-H6-1 append-only; D-H6-4 erasure; R98; R82
- **File + line:** `src/audit-log/audit-log.service.ts:77-91`; `prisma/migrations/20261226000000_create_audit_log/migration.sql:74-107`; `test/audit-log/audit-log.migration.spec.ts:115-122`
- **Observation:** `AuditLogService.redactPii` updates `auditLogEntry` rows in place, while the migration enables FORCE RLS, creates no UPDATE policy, and revokes UPDATE/DELETE from `PUBLIC` and `app_runtime`. The tests assert “NO UPDATE or DELETE policy” but do not prove the erasure method can actually execute under the intended runtime principal.
- **Required fix:** Make the privilege model explicit and tested: either add a narrowly-scoped privileged erasure role/path with UPDATE allowed only for tokenization and prove it with role-switched integration tests, or remove in-place UPDATE and redesign the erasure mechanism with an operator-approved exception. Do not claim both “no runtime principal can update” and “the service updates rows” without a tested privilege boundary.
- **Evidence:** The service calls `this.prisma.auditLogEntry.update` at `audit-log.service.ts:84-90`; the migration creates no UPDATE policy at `migration.sql:74-99` and revokes UPDATE/DELETE at `migration.sql:107-108`; the static test explicitly expects no UPDATE/DELETE policy at `audit-log.migration.spec.ts:115-122`.

### F4 — write-path PII redaction uses a fixed sentinel, not the mandated deterministic erasure token
- **Severity:** BLOCKER
- **Doctrine cite:** R98; D-H6-4; H6A builder brief lines 50–80
- **File + line:** `src/audit-log/erasure-token.ts:28`; `src/audit-log/erasure-token.ts:107-115`; `src/audit-log/audit-log.service.ts:51-52`; `test/audit-log/audit-log.service.spec.ts:412-428`
- **Observation:** The write path replaces PII leaves with the constant `[REDACTED:GDPR-ART-17]`. The H6A brief says audit_log before/after PII must be tokenized via `erasure-token.ts`, and the token primitive is specifically deterministic HMAC `tok_<16 hex>` so redacted audit rows remain correlatable without plaintext. This implementation discards correlation on initial audit writes and only HMAC-tokenizes later during explicit erasure.
- **Required fix:** Use `erasureToken(String(value))` for write-path PII leaves unless there is an operator-approved reason to store a non-correlatable sentinel. Update tests to expect `tok_[a-f0-9]{16}`, not `ERASURE_TOKEN`, for before_state/after_state PII.
- **Evidence:** `ERASURE_TOKEN` is defined as a fixed string at `erasure-token.ts:28`, `redactState` writes that constant at `erasure-token.ts:107-115`, and the service writes `redactPii(ctx)` into the audit row at `audit-log.service.ts:51-52`; the test asserts the fixed sentinel at `audit-log.service.spec.ts:412-428`.
