## P0 findings
- [src/screens/coach/payments/CoachPackageContentsScreen.tsx:325-343] Fast double-tap on Confirm can dispatch two `coachPackageContentsApi.push` calls, and the two calls can receive different idempotency keys. The guard is React state (`pushSubmitting`) and the idempotency key is also React state (`pushIdemKey`); `setPushSubmitting(true)` and `setPushIdemKey(key)` are not synchronously visible to a second press handler that fires before the component re-renders, so that second handler still sees `pushSubmitting === false` and `pushIdemKey === null`, generates another UUID, and calls `push` again. This is P0 because the protected operation is a buyer push side effect and the duplicate can bypass backend idempotency by using a different `Idempotency-Key`. Concrete fix: use a synchronous ref/lock for the in-flight guard and a ref-backed idempotency key assigned before awaiting the API call (for example, `pushInFlightRef.current` and `pushIdemKeyRef.current`), reset those refs only on cancel/success/new intent, and add a test that presses Confirm twice back-to-back before waiting for the submitting render.

## P2 findings
- [src/screens/coach/payments/CoachPackageContentsScreen.tsx:281-295] The preview step has no loading affordance: `onPushExisting` closes the prompt and then awaits `pushPreview`, but there is no `previewLoading` state, inline spinner, disabled row state, or loading posture before the confirm modal appears. This violates the M5 contract requirement to show a calm inline state while preview is loading and not leave a dead tap. Concrete fix: keep visible feedback during `pushPreview` (for example a sheet loading row or modal loading state), block repeated preview taps while it is in flight, and test the loading state.
