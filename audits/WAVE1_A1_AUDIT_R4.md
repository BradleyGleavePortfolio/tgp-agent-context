## P1 findings
- [src/ai/ai.service.ts:61-99, src/ai/ai.service.ts:112-138, src/ai/ai.service.ts:396-429, src/ai/ai.service.ts:458-468, src/ai/ai.service.ts:510-528, src/ai/ai.service.ts:755-766] The daily cap is still not a provable hard pre-spend bound on provider TOTAL tokens because `MAX_INPUT_CHARS / 3` is not a tokenizer upper bound for Perplexity/Anthropic input. The code reserves a fixed 6,600 tokens on the assumption that 18,000 UTF-16 JS characters can never tokenize above 6,000 input tokens, but that assumption is false for valid user input: CJK text, many emoji/symbol/rare Unicode sequences, and even many short-token patterns can tokenize at well above 1 token per 3 characters. A request consisting of high-token-density content can therefore pass the 6,600-token reservation, reach the provider, and bill more than the reserved worst case; the new `Math.min(actual, reserved)` reconcile then clamps the ledger down to the reservation instead of recording the provider-reported larger total, masking the overspend rather than preventing it. Fix: replace the chars/3 proof with a real provider-tokenizer bound or an actually safe byte/character ceiling for the worst token density accepted by both providers, and reject or truncate based on that bound before reserving/calling the provider.
- [src/ai/ai.service.ts:101-138, src/ai/ai.service.ts:396-403, src/ai/ai.service.ts:455-464, src/ai/ai.service.ts:510-521] `clampPromptParts()` does not clamp every byte/character that becomes provider input, so even its character ceiling is not actually the provider-input ceiling claimed in comments/tests. The function preserves the whole `systemPrompt` even if it alone exceeds `MAX_INPUT_CHARS`, only budgets `entry.content` and `userMessage`, and does not account for Anthropic wrapper text (`Assistant: `, `User: `, newlines) or provider message-role framing. A saturated tail can therefore produce provider input above `MAX_INPUT_CHARS` even before tokenization, and an oversized rendered system/context block bypasses the clamp entirely. Fix: construct the exact provider payload first (including system/context, role labels/framing, separators, and final user turn), then clamp/reject that exact payload against the same proven pre-spend bound used for reservation.
