### N1 (P1) — UUID migration silently destroys non-empty wearable-prompt data instead of aborting into #404/backfill

**File:** `prisma/migrations/20261219000000_community_wearable_prompts_uuid_id/migration.sql:17-19,36-38`

```sql
--   If a future environment is found to hold rows in these tables, this migration
--   MUST be revisited with a data backfill before applying — see the R82 tracking
--   issue referenced in PR #399's R81 cleanup.
...
-- Drop child first (FK), then parent. IF EXISTS keeps the migration idempotent.
DROP TABLE IF EXISTS "community_wearable_prompt_sources";
DROP TABLE IF EXISTS "community_wearable_prompts";
```

The migration documents the exact dangerous condition — a non-empty environment must take the backfill path — but the executable SQL never checks for that condition. If either table contains rows, the migration does not redirect to the tracking issue and does not fail loudly before damage; it drops the child table and parent table. That is the opposite of the required non-empty-environment guard.

`IF EXISTS` also does not make this safely idempotent in the operational sense. It prevents an error if the table is absent, but a repeated manual run after data exists in the recreated UUID table would still drop the data again.

**Repro:**
1. On a DB that has applied PR #399's original `20261217000200_community_wearable_prompts`, insert one prompt row and one source row.
2. Apply `20261219000000_community_wearable_prompts_uuid_id/migration.sql`.
3. The migration executes `DROP TABLE IF EXISTS` for both tables before recreating them; the rows are gone instead of the migration aborting with a backfill/#404 instruction.

**Fix prescription:** Add an executable preflight before either `DROP TABLE`, inside the same transaction. Example shape:

```sql
DO $$
BEGIN
  IF to_regclass('public.community_wearable_prompt_sources') IS NOT NULL
     AND EXISTS (SELECT 1 FROM "community_wearable_prompt_sources" LIMIT 1) THEN
    RAISE EXCEPTION 'community_wearable_prompt_sources is non-empty; stop and follow GitHub issue #404 backfill path before applying 20261219000000';
  END IF;

  IF to_regclass('public.community_wearable_prompts') IS NOT NULL
     AND EXISTS (SELECT 1 FROM "community_wearable_prompts" LIMIT 1) THEN
    RAISE EXCEPTION 'community_wearable_prompts is non-empty; stop and follow GitHub issue #404 backfill path before applying 20261219000000';
  END IF;
END $$;
```

Also update the static schema/migration spec to assert the guard exists and explicitly names `#404`.

### N2 (P2) — `Fixes #404` will auto-close the deferred tracking issue on merge

**File:** PR body, Tracking section

```md
- Fixes #404 — community wearable-prompts cuid→UUID re-key needs a data backfill if any non-empty environment is ever detected (the F1 DROP+RECREATE is empty-table-only; Decision 8 (A)).
```

Issue #404 is open and is explicitly a conditional tracking issue for the future/non-empty-environment backfill path. The issue body says the backfill work is required only if rows are ever found and includes unchecked backend/data-ops tasks. Closing it when PR #405 merges would violate R82 because the durable tracker for deferred/out-of-lane work disappears before the condition it tracks has been resolved.

The PR body also claims the migration header references #404, but the migration header only says “see the R82 tracking issue referenced in PR #399's R81 cleanup” and does not include `#404` or the issue URL (`migration.sql:17-19`). The issue body similarly claims the migration header explicitly references the issue, which is not true in the audited file.

**Repro:** Merge PR #405 as written; GitHub will auto-close #404 because the PR body contains `Fixes #404`, even though #404's checklist is still intentionally open for future non-empty-environment handling.

**Fix prescription:** Change the PR body language to `Refs #404` / `Tracks #404`, not `Fixes #404`. Update the migration header to explicitly name `#404` and the issue URL. Leave #404 open until either (a) every environment is verified empty and the team intentionally closes it with evidence, or (b) a real non-empty backfill migration is shipped and audited.

### N3 (P2) — F4 cooldown is not service-clock-injected or timer-mocked

**File:** `src/community/wearable-prompts/wearable-prompts.service.ts:76,187,195`; `src/community/wearable-prompts/__tests__/wearable-prompts.cooldown.spec.ts:37-77`

```ts
const now = new Date();
...
const fresh = await this.repo.markDismissed(promptId, coach.id, new Date());
...
const fresh = await this.repo.markActedOn(promptId, coach.id, new Date());
```

The repository-level cooldown method accepts an explicit `now`, which is good, but the production service still constructs the clock inline. The new cooldown test proves the repository WHERE clause ignores `dismissedAt`; it does not use `jest.useFakeTimers()` / `jest.setSystemTime()` and does not prove the service-level `generate` path passes the mocked current time into `isWithinCooldown` at the 24h boundary.

This misses the re-audit requirement for “service-level guard with clock injection” and “tests with timer mocking.” Under R79, the exact shipped contract should be regression-pinned at the handler/service seam, not only at a lower repository helper.

**Repro:** Remove the `now` capture from `generate` and pass `new Date(0)` or a stale constant into `repo.isWithinCooldown`; the current repository test still passes because it calls `isWithinCooldown` directly, and the service test only stubs `isWithinCooldown` as a boolean.

**Fix prescription:** Inject a small clock provider (or constructor param defaulting to `() => new Date()`) into `WearablePromptsService`, use it for `generate`, `dismiss`, and `actOn`, and add a service spec with `jest.useFakeTimers().setSystemTime(...)` proving `repo.isWithinCooldown` receives the mocked time and respects the 24h boundary.

### N4 (P3) — response Zod schemas did not pin the new UUID id contract

**File:** `src/community/wearable-prompts/wearable-prompts.dto.ts:67-91`

```ts
export const PromptSourceViewSchema = z
  .object({
    sampleId: z.string(),
    metricKey: z.string(),
    observedValue: z.number(),
  })
  .strict();
...
export const PromptViewSchema = z
  .object({
    id: z.string(),
    workspaceId: z.string(),
    coachId: z.string(),
    clientId: z.string(),
```

The schemas are `.strict()`, but the F1 public contract changed prompt ids to UUIDs and the response validator still accepts any string for `PromptViewSchema.id`. That means `GenerateResponseSchema.parse()` / `PromptListResponseSchema.parse()` would not catch a regression that emitted a cuid or other non-UUID prompt id, despite the controller path params requiring UUID v4.

**Repro:** In a unit test, set `persistedPrompt().id = 'clabc123'`; `toView` returns that id and `GenerateResponseSchema.parse()` accepts it because `id` is only `z.string()`.

**Fix prescription:** Update `PromptViewSchema.id`, `workspaceId`, `coachId`, `clientId`, and `PromptSourceViewSchema.sampleId` to `z.guid(...)` / UUID validators (or at least `id` for the F1 prompt-id contract). Add a regression test that a cuid prompt id fails response parsing.
