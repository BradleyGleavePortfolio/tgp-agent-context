## P0 findings
- [src/billing/billing.service.ts:1418] `applyPayoutFailed` uses `updateMany` with `where: { id: row.id, NOT: { last_payout_stripe_id: payout.id, last_payout_status: terminalStatus } }` to both record the failure and decide whether to send the COACH_ALERT. [prisma/schema.prisma:3666] and [prisma/schema.prisma:3668] make both guarded fields nullable, so a snapshot with no prior payout values produces SQL equivalent to `NOT (last_payout_stripe_id = ? AND last_payout_status = ?)`; with both columns NULL this predicate is UNKNOWN, not TRUE, so the first `payout.failed`/`payout.canceled` can update 0 rows. Because the handler then returns before alerting at [src/billing/billing.service.ts:1437] and the outer webhook still marks the event complete at [src/billing/billing.service.ts:506], Stripe replay is dedup-swallowed and the failed bank payout is never persisted or surfaced. I reproduced this with Prisma 6.19.3 in a temporary nullable-field model: the generated `UPDATE ... WHERE ... NOT (nullable_col = ? AND nullable_col2 = ?)` returned `{ count: 0 }` for an all-NULL row. This is the original B7 money/ops failure still present for first-payout/no-snapshot-values cases. Concrete fix: after `findFirst`, compare `row.last_payout_stripe_id === payout.id && row.last_payout_status === terminalStatus` in TypeScript and return only for that exact already-terminal state; otherwise update by `{ id: row.id }` without nullable `NOT`, or make the Prisma predicate explicitly NULL-safe.
