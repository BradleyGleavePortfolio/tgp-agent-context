## P0 findings
- [src/coach/command-center/ltv-metrics.controller.ts:20, src/coach/command-center/ltv-metrics.controller.ts:35, src/coach/command-center/ltv-metrics.controller.ts:57] Active sub-coaches can still reach the financial LTV metrics route. The builder's SC-1 safety claim says the financial surface keeps `NoActiveSubCoachGuard`, but `LtvMetricsController` is guarded only by `JwtAuthGuard` and `CoachGuard`; `CoachGuard` admits any `role === 'coach'`, and the handler returns the LTV metrics payload. This is a P0 financial-data exposure. Concrete fix: add `NoActiveSubCoachGuard` to `LtvMetricsController` (class or `GET ltv-metrics` handler) and add a controller test proving an active sub-coach is denied while head coaches/owners still pass.

## P1 findings
- [src/coach/command-center/command-center.service.ts:393, src/admin/ptm/admin-ptm.service.ts:425, src/admin/ptm/admin-ptm.service.ts:427, src/coach/command-center/command-center.service.ts:402] `getAtRisk()` is still over-restricted for sub-coaches. The new code calls `adminPtm.getRiskBoardForCoach(coachId, ...)` before applying the resolved SubCoachScope intersection, but `AdminPtmService.getRiskBoardForCoach()` first builds its board from `User.coach_id = coachId`. For a sub-coach, assigned students remain under the head coach's `coach_id`, so the board is empty before the intersection and assigned at-risk clients never appear. Concrete fix: build the at-risk list directly from `resolveScope()` client ids, or call the risk-board path with the head/owner coach id and then intersect with assigned client ids.
- [src/coach/command-center/command-center.service.ts:710, src/coach/command-center/command-center.service.ts:753, src/coach/coach-alerts.service.ts:149, src/coach/coach-alerts.service.ts:150] Sub-coaches can list assigned alerts but cannot dismiss them. `getActionQueue()` correctly reads head-owned alert rows with `(coach_id = ownerCoachId, client_id IN assigned)`, but `dismissAlert()` still passes the sub-coach id into `CoachAlertsService.acknowledge()`, whose conditional update requires `coach_id = coachId`. A sub-coach therefore receives a visible action item and then gets `NotFoundException` when dismissing it. Concrete fix: resolve scope in `dismissAlert()`, verify the alert's `client_id` is in the authorized client set, and acknowledge using `ownerCoachId` (or add an alert-service method that atomically updates by `id + ownerCoachId + client_id IN assigned`).
- [src/coach/command-center/command-center.service.ts:643, src/coach/command-center/command-center.service.ts:646, src/coach/command-center/command-center.service.ts:649, src/coach/command-center/command-center.service.ts:664, src/messaging/messaging.service.ts:427, src/messaging/messaging.service.ts:433, src/messaging/messaging.service.ts:435] Sub-coach inbox unread/turn state is wrong for messages sent by the sub-coach. Messaging stores a sub-coach's outgoing row under the head coach's `coach_id` while preserving the actual sub-coach as `sender_id`; Command Center now filters unread with `NOT: { sender_id: ownerCoachId }` and computes `is_coach_turn` as `latestMsg.sender_id !== ownerCoachId`. For a sub-coach caller, their own outgoing message has `sender_id = subCoachId`, so it is counted as unread and shown as the coach's turn. Concrete fix: preserve `threadCoachId/ownerCoachId` for the `coach_id` namespace but use the caller id (or stricter `sender_id = client_id`) for unread and turn-taking semantics; add a sub-coach inbox test with an outgoing sub-coach message.
