### F2 (P2) ŌĆö `HabitsScreen` extracts `coach_reviewed_at` via untyped `as`-cast from `useQuery<unknown>`

**File:** `src/screens/client/HabitsScreen.tsx:90-101`

```ts
const row = todayCheckInQ.data as
  | {
      mood?: number;
      energy?: number;
      sleep_hours?: number;
      sleep_quality?: number;
      stress?: number;
      notes?: string;
      date?: string;
      coach_reviewed_at?: string | null;   // ŌåÉ new field added by this PR
    }
  | null
  | undefined;
```

`useTodayCheckIn` returns `useQuery<unknown>` (confirmed at `src/hooks/useApi.ts:406`). The pre-existing fields (`mood`, `energy`, `sleep_hours`, etc.) are extracted through the same cast, which is pre-existing debt. However, **this PR specifically adds** `coach_reviewed_at?: string | null` to this cast and then consumes it at line 107:

```ts
setCoachReviewedAt(row.coach_reviewed_at ?? null);
```

There is no runtime validation that the backend is actually returning a `string` for `coach_reviewed_at`. If the backend returns the field as a `Date` object (possible in some serialization paths) or if the field is renamed (e.g., to `reviewedAt` during a future schema change), the `?? null` coalesce silently yields `null` ŌĆö the pill never renders, there is no error, and the failure is invisible. The backend PR #398 F2 established that `markReviewedByCoach` returns the raw Prisma `CheckIn` row without a Zod envelope; while this mobile PR does not call that endpoint, it does rely on `checkInsApi.list` which also returns raw Prisma rows (same architectural gap ŌĆö the check-in list endpoint has no narrowed response envelope in backend scope). The `coach_reviewed_at` value that reaches the mobile is whatever Prisma serializes.

This contrasts with `MessagesScreen`'s consumption of `coachReviewedAt`:

```ts
const res = await messagesApi.coachReview();
setCoachReviewedAt(res.data?.coachReviewedAt ?? null);
```

There, the API type annotation `api.get<{ coachReviewedAt: string | null }>` explicitly names the field, making a rename visible at the TypeScript layer (compile-time warning). In `HabitsScreen`, the extraction is from a `useQuery<unknown>` cast where the field name `coach_reviewed_at` could silently drift.

Severity P2 (not P1): the field is currently returning correctly (the backend `CheckIn.coach_reviewed_at` is a `DateTime?` that Prisma serializes as ISO string or null), and the pill failing silently to render is degraded-but-not-broken UX (the pill stays hidden). However, this is the same "relies on the Prisma row shape without a narrowed response contract" gap that PR #398 F2 identified as a must-fix, and the mobile side compounds it.

**Backend reference:** PR #398 F2 ŌĆö `markReviewedByCoach` returns full raw `CheckIn` row. The `check-ins` list endpoint (called by `useTodayCheckIn`) is governed by the same `checkInsApi.list` which has no Zod envelope; the `coach_reviewed_at` field is silently added to the backend `CheckIn` model by the backend PR's migration. The mobile must tolerate the unbounded row shape without `.strict()`-style rejection (it does ŌĆö no Zod at all in this path), but it also must be resilient to field absence.

**Recommended fix:** Two options, in order of preference:
1. Narrow the `useTodayCheckIn` hook return type to a typed interface that includes `coach_reviewed_at?: string | null`, or add a safe accessor with a runtime check: `const coachReviewedAt = typeof row?.coach_reviewed_at === 'string' ? row.coach_reviewed_at : null;`. This guards against a non-string value at runtime without requiring a Zod dependency.
2. As a minimum: document the coupling explicitly (a comment noting "field name is `coach_reviewed_at` per the Prisma `CheckIn` model ŌĆö any backend rename requires updating this cast").

### F3 (P3) ŌĆö `earlier today` branch in `romanCoachReviewRelative` is dead code

**File:** `src/lib/roman/copy.ts:599`

```ts
export function romanCoachReviewRelative(reviewedAt: string, now: Date = new Date()): string {
  const then = new Date(reviewedAt);
  const diffMs = now.getTime() - then.getTime();

  if (diffMs < MS_PER_HOUR) return 'just now';

  const dayDiff = localDayIndex(now) - localDayIndex(then);

  if (diffMs < 24 * MS_PER_HOUR && dayDiff === 0) {
    const hours = Math.floor(diffMs / MS_PER_HOUR);
    return hours === 1 ? '1 hour ago' : `${hours} hours ago`;
  }

  if (dayDiff === 0) return 'earlier today';  // ŌåÉ unreachable
  if (dayDiff === 1) return 'yesterday';
  if (dayDiff <= 7) return `${dayDiff} days ago`;
  return `on ${MONTHS[then.getMonth()]} ${then.getDate()}`;
}
```

The `earlier today` branch at line 599 is unreachable by construction. It requires both `diffMs >= 24 * MS_PER_HOUR` (more than 24 hours of wall-clock separation) and `dayDiff === 0` (same `localDayIndex`). Since `localDayIndex` computes `year * 372 + month * 31 + day` using local calendar time, `dayDiff === 0` means both timestamps fall on the same local calendar day. The maximum wall-clock separation between two timestamps on the same local calendar day is 23 hours 59 minutes 59 seconds, which is strictly less than 24 hours. Therefore `diffMs >= 24h` and `dayDiff === 0` cannot both be true simultaneously. The preceding branch consumes the only reachable case where `dayDiff === 0` (same day, within 24h ŌåÆ hours ago), so the `earlier today` return is dead code.

The doc-comment at the top of `romanCoachReviewRelative` lists `"same calendar day ŌåÆ 'earlier today'"` as a bucket, creating a mismatch between specification and implementation. The `CompetencePill.test.tsx` comment at line 8 also lists `"earlier today"` as a bucket but provides no test for it (no test can exercise it given the constraint above).

Severity P3: no user-visible defect (the bucket is never reached), but the dead branch adds misleading documentation and could confuse a future maintainer who tries to write a test for it.

**Recommended fix:** Remove the `if (dayDiff === 0) return 'earlier today';` line (it is unreachable). Update the doc-comment to remove the `"same calendar day ŌåÆ 'earlier today'"` bucket entry. The effective bucket table after the fix is: `< 1 hour ŌåÆ just now`, `< 24h same day ŌåÆ N hour(s) ago`, `yesterday`, `2ŌĆō7 days ago`, `older ŌåÆ Month D`.
