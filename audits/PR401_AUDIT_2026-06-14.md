The CI failure is a **real defect in this PR's own code** (not infrastructure). P3: `getRegimeRevisions` has no `take` cap at the query level (bounded only by the eviction window, which is operator-configurable).

## Per-finding detail

### F1 (P1) — 5-module directed cycle breaks module-graph.spec.ts gate (CI failure root cause)

**File:** `src/regimes/regimes.module.ts:36`

```ts
// regimes.module.ts
imports: [forwardRef(() => AuthModule), forwardRef(() => PackagesModule)],
```

The PR introduces this import graph edge: `RegimesModule → AuthModule`. Combined with the pre-existing path `AuthModule → InviteCodesModule → BillingModule → CheckoutModule`, and the PR's new edge `CheckoutModule → RegimesModule` (via `forwardRef(() => RegimesModule)` in `checkout.module.ts`), the complete 5-node directed cycle is:

```
AuthModule → InviteCodesModule → BillingModule → CheckoutModule → RegimesModule → AuthModule
```

`test/module-graph.spec.ts` explicitly allows only 2-node cycles (length-3 arrays `[A, B, A]`) in `KNOWN_FORWARDREF_CYCLES`. Its `isKnownForwardRefCycle()` function:

```ts
// test/module-graph.spec.ts
function isKnownForwardRefCycle(cycle: readonly string[]): boolean {
  if (cycle.length !== 3) return false;   // ← rejects this 5-node (length-6) cycle
  ...
}
```

The CI logs confirm the exact failure:

```
FAIL test/module-graph.spec.ts
  ● AppModule dependency graph › has no unexpected directed import cycles
    Found 1 disallowed module dependency cycle(s).
    Cycles:
      [1] AuthModule → InviteCodesModule → BillingModule → CheckoutModule → RegimesModule → AuthModule
```

`test/openapi-spec.spec.ts` also fails with a 20 s `beforeAll` timeout attempting to compile `AppModule`. The same cycle causes the slower init (Nest DI scanning the circular import graph under forwardRef takes significantly longer than a DAG compile, and the 20 s budget is insufficient for this PR's module count).

**Recommended fix:** Remove `AuthModule` from `RegimesModule`'s imports. `JwtAuthGuard` and `RolesGuard` can be moved to the repo's `SecurityGuardsModule` (or another `@Global` module with no feature-module imports) — the established pattern for breaking auth-guard cycles documented in the spec's option (a). `RolesGuard` is already provided locally in `RegimesModule`; `JwtAuthGuard` resolves from `AuthModule`. If `JwtAuthGuard` is already global (check `SecurityGuardsModule` or the `@Global` annotation on `AuthModule`), the explicit `AuthModule` import is unnecessary and can simply be dropped.

### F2 (P1) — `onPartialRefund` TOCTOU: findUnique+create without tx or P2002 catch

**File:** `src/regimes/partial-refund-decision.service.ts:84–96`

```ts
// onPartialRefund — no wrapping tx, no P2002 catch:
const existing = await db.partialRefundDecision.findUnique({
  where: { stripe_refund_id: args.stripe_refund_id },
  select: { id: true },
});
if (existing) return false;

await db.partialRefundDecision.create({          // ← uncaught P2002 on race
  data: {
    client_purchase_id: args.client_purchase_id,
    stripe_refund_id: args.stripe_refund_id,
    decision: 'pending',
  },
});
```

Under concurrent Stripe redeliveries, two webhook invocations can both execute `findUnique` before either executes `create`. Both observe `null`. The first `create` succeeds; the second hits the `@unique` constraint on `stripe_refund_id` and raises a Prisma `P2002` error. Because there is no `try/catch` around the create, the error propagates through `onChargeRefunded` to the webhook handler, which will return a 500 and cause Stripe to retry — or, if the outer handler swallows the error, silently skip the alert emit that follows.

The established pattern in the same file (`upsertAndApplyRefund`, lines 641–714) handles this correctly:

```ts
// Existing pattern — correct:
try {
  await tx.chargeRefund.create({ ... });
  isFirstObservation = true;
} catch (err) {
  if ((err as { code?: string }).code !== 'P2002') throw err;
  isFirstObservation = false;
}
```

`onPartialRefund` deviates without justification.

**Recommended fix:** Wrap the `create` in a try/catch that swallows `P2002` and returns `false` (mirrors the dispute pattern). Alternatively, use Prisma's `createMany` with `skipDuplicates: true` (but this doesn't return the created row). The try/catch approach is preferred for consistency:

```ts
try {
  await db.partialRefundDecision.create({ data: { ... } });
} catch (err) {
  if ((err as { code?: string }).code === 'P2002') return false;
  throw err;
}
return true;
```

The `findUnique` pre-check can be removed once the catch handles P2002, making the operation a true upsert-noop on collision.

### F3 (P2) — `PartialRefundDecision` table created without RLS

**File:** `prisma/migrations/20261214000000_named_regimes_and_partial_refund_decision/migration.sql`

The migration creates `PartialRefundDecision` with a FK, two indexes, and a PK — but omits:

```sql
-- MISSING in migration:
ALTER TABLE "PartialRefundDecision" ENABLE ROW LEVEL SECURITY;
ALTER TABLE "PartialRefundDecision" FORCE ROW LEVEL SECURITY;
CREATE POLICY "p_partialrefunddecision_service_role_all" ON "PartialRefundDecision" ...
CREATE POLICY "p_partialrefunddecision_coach_all" ON "PartialRefundDecision" ...
```

Every new table holding financial data in this codebase has RLS. `ChargeRefund` (the analogous financial record) has `ENABLE` + `FORCE` + `service_role` bypass + `purchase-party-read` policies in `20261213000000_rls_tier1_phi_financial_privacy/migration.sql`. The `rls-floor-guard` job passes because that check is currently soft (`RLS_ENFORCEMENT_FULL=off`), but this table will fail the hard floor guard once that flag flips, and in the meantime it bypasses row-level isolation in the Supabase client layer.

**Recommended fix:** Add the following to the migration (before the FK constraint):

```sql
ALTER TABLE "PartialRefundDecision" ENABLE ROW LEVEL SECURITY;
ALTER TABLE "PartialRefundDecision" FORCE ROW LEVEL SECURITY;

CREATE POLICY "p_partialrefunddecision_service_role_all"
  ON "PartialRefundDecision" AS PERMISSIVE FOR ALL TO service_role
  USING (true) WITH CHECK (true);

CREATE POLICY "p_partialrefunddecision_coach_all"
  ON "PartialRefundDecision" AS PERMISSIVE FOR ALL TO public
  USING (
    EXISTS (
      SELECT 1 FROM public."ClientPurchase" cp
      WHERE cp."id" = "PartialRefundDecision"."client_purchase_id"
        AND cp."coach_user_id" = app.current_user_id()
    )
  )
  WITH CHECK (
    EXISTS (
      SELECT 1 FROM public."ClientPurchase" cp
      WHERE cp."id" = "PartialRefundDecision"."client_purchase_id"
        AND cp."coach_user_id" = app.current_user_id()
    )
  );
```

Client SELECT access is intentionally omitted (the coach-decision surface is coach-only); the service role bypass covers webhook-driven inserts.

### F4 (P2) — Write routes lack `@Throttle` decorator

**Files:** `src/regimes/regimes.controller.ts` (three POST/PATCH handlers), `src/regimes/refund-decisions.controller.ts` (one POST handler)

The following mutating routes carry no explicit `@Throttle`:

- `POST /coach/regimes/:id/promote-from-program` — flips `is_regime=true`, has irreversible downstream effects on package-content attachment rules.
- `POST /coach/regimes/:id/archive` — archives a regime, blocks future attachments.
- `PATCH /coach/regimes/:id` — writes a `WorkoutProgramRevision` row and triggers eviction.
- `POST /coach/refunds/:refundId/decide` — the most critical: applies a financial decision that calls `cancelPendingForPurchase` (cancels `ScheduledDrop` rows).

The existing pattern applies `@Throttle` to every write route that has downstream financial impact (`src/checkout/checkout.controller.ts:95,126`) or write amplification risk. The global throttler provides a 300 req/min default for authenticated routes, which is insufficient protection for a route that triggers irreversible `ScheduledDrop` cancellation.

**Recommended fix:**

```ts
// refund-decisions.controller.ts
@Throttle({ default: { limit: 10, ttl: 60000 } })
@Post(':refundId/decide')
async decide(...) { ... }

// regimes.controller.ts — on the three write handlers
@Throttle({ default: { limit: 30, ttl: 60000 } })
@Post(':id/promote-from-program')
async promote(...) { ... }
```

### F5 (P3) — `getRegimeRevisions` has no query-level `take` cap

**File:** `src/regimes/regimes.service.ts:142`

```ts
const revisions = await this.prisma.workoutProgramRevision.findMany({
  where: { program_id: id },
  orderBy: { revision_index: 'desc' },
  select: { revision_index: true, created_at: true, cause: true },
  // no take: N
});
```

Under default configuration (`revision_retention_count=3`), the eviction service keeps at most 3 rows — so the effective bound is low. However, `revision_retention_count` is an operator-configurable column; if an operator increases it to (e.g.) 100, this query returns 100 rows without pagination. The query is protected by the per-route auth chain but the database round-trip is unbounded at the SQL level.

**Recommended fix:** Add `take: program.revision_retention_count` (available from `requireOwnedRegime`) or a constant cap matching the designed UX ("last 3 versions" drawer):

```ts
const revisions = await this.prisma.workoutProgramRevision.findMany({
  where: { program_id: id },
  orderBy: { revision_index: 'desc' },
  take: 20,   // generous cap; production default is 3
  select: { revision_index: true, created_at: true, cause: true },
});
```

## What's correctly implemented (do not regress)

- **Flag-off default-OFF architecture:** `isNamedRegimesEnabled()` reads `process.env` per call (never boot-cached), returns ON only for exact case-insensitive `'true'`. `NamedRegimesFeatureGuard` returns 404 (not 403) while the flag is OFF — correct "feature doesn't exist" posture (#5 server-authoritative). The partial-refund hook also re-checks the flag before writing, so no decision rows are created while the feature is hidden.
- **Class-level `@Roles('coach')` on both controllers:** `RegimesController` and `RefundDecisionsController` both carry class-level `@Roles('coach')` and `@UseGuards(JwtAuthGuard, RolesGuard, NamedRegimesFeatureGuard)`. The `regimes-roles-pin.spec.ts` spec pins this via `Reflect.getMetadata`, so accidental removal fails CI. No `LEGACY_GUARD_ALLOWLIST` entry required.
- **Cross-tenant 404-not-403:** `requireOwnedRegime` uses `findFirst({ where: { id, coach_id: coachId, is_regime: true } })` and throws `NotFoundException` — a cross-tenant regime is indistinguishable from a non-existent one. `decide()` in `PartialRefundDecisionService` checks `row.client_purchase.coach_user_id !== coachUserId` and throws `NotFoundException` (not `ForbiddenException`). Both are correct per 50-failures #11.
- **TX safety in `decide()`:** The decision `updateMany` and `cancelPendingForPurchase` run in a single `$transaction`. The WHERE-guard `decision='pending'` makes the update idempotent under concurrent second decide calls.
- **Additive migration with FK:** All three `WorkoutProgram` columns have safe defaults (`false`, `null`, `3`); existing rows are unaffected. The `PartialRefundDecision` FK uses `ON DELETE CASCADE` — consistent with the pre-existing `ChargeRefund.purchase_id` FK pattern.
- **R0 clean:** No `@ts-ignore`, `as any`, `as unknown as`, `.catch(()=>undefined)`, or `Coming soon` in any new production source line.
- **No assistant-attribution commit trailers:** All commits authored by `Bradley Gleave <bradley@bradleytgpcoaching.com>`.
- **`promoteFromProgram` and `archiveRegime` idempotency:** `updateMany` with WHERE guards means a second promote/archive call is a no-op flip.
- **Regime-archived 422 guard:** `package-contents.service.ts` correctly returns 422 (not 404) on an archived regime — the asset exists, it's simply not attachable.
- **Revision-retention eviction:** `evictForRegime` deletes by `revision_index < threshold` (idempotent; threshold ≤ 0 short-circuits to 0 deleted). Clamps `retention = Math.max(1, ...)` so no configuration can wipe the entire history.
- **`ParseUUIDPipe` on all route params that accept a UUID:** `RegimesController` applies `ParseUUIDPipe` on `:id` parameters, rejecting malformed IDs at the controller boundary.
- **Batch groupBy for attachment counts:** `listRegimes` uses a single `groupBy` query (not N individual `count` calls) for attachment counts.
