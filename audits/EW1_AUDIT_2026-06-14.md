## 4. Dependency Graph

### 4.1 Coach Workout Builder Consumers of Exercise Endpoints

The **existing** `CoachWorkoutBuilderScreen` calls `GET /exercises/search` (Module A / ExerciseDB proxy) via `useExerciseSearch`. EW1 work on Module A (e.g. adding difficulty filter, fixing combined-param logic) is **safe** — the endpoint signature is additive (new optional query params), and the existing call passes only `{ q, limit: 8 }` so it is immune to new params.

`WorkoutPlanExercise.exercise_external_id` is a loose string — it can store `seed:push-001`, an ExerciseDB id, or an `ExerciseCatalogItem.id/slug`. EW1 work adding a `custom_exercise_id` column would NOT conflict with the existing string column; it would be a new nullable FK alongside it. **No existing code breaks** from additive schema work.

### 4.2 EW2 Cross-Lane Risk (EW2 undo button, PR #253)

EW2 is purely client-side (`EXPO_PUBLIC_FF_MWB_UNDO` default OFF). It operates on the existing `useWorkoutBuilder` mutations (`addExercise`, `removeExercise`, `reorderExercise`, `editExerciseField`) by wrapping them with a command stack. EW2 **does not touch** any exercise-library or exercise-catalog code path. Risk: **low/zero** for EW1 backend work. For EW1 mobile work that **modifies** `CoachWorkoutBuilderScreen.tsx`, care is needed not to change the mutation call signatures that EW2's `actionForEditExerciseField` depends on:

```typescript
// useWorkoutBuilder.ts:107-112
export function actionForEditExerciseField(
  ...) { return { kind: 'editExerciseField', clientId, field, previousValue }; }
```

The `editExerciseField` inverse-op captures `clientId` + `field` + `previousValue`. The `DraftExerciseRow` interface (fields `sets`, `reps_or_duration_seconds`, `rest_seconds`, `weight_lbs`, `superset_group_id`, `notes`) is the row shape EW2 assumes. **EW1c (custom exercise CRUD)** must NOT rename or restructure these fields, or the EW2 inverse-op map (`CoachWorkoutBuilderScreen.tsx:241–258`) will silently stop tracking those fields. Coordinate: if EW1c adds fields to `DraftExerciseRow`, corresponding entries in the EW2 inverse-op map must be added (separate PR, flagged).

### 4.3 L13 `editExerciseField` Row Shape

The `FIELD_TO_SNAPSHOT` and `SNAPSHOT_TO_FIELD` maps in `CoachWorkoutBuilderScreen.tsx:241–264` enumerate every tracked field. Currently these are: `sets`, `reps_or_duration_seconds`, `rest_seconds`, `weight_lbs`, `superset_group_id`, `notes`. If EW1 (custom exercise CRUD) adds a `custom_label` override field or a `source_catalog_id` reference to `DraftExerciseRow`, it must also add those to the EW2 map — otherwise undo of an edit to that field silently no-ops. This is a **coordination requirement**, not a runtime breakage.

### 4.4 Cross-Lane Risk with D MWB Visual Refactor

No `D_MWB` or "visual refactor" branch/commit/spec was found in either repo (`grep -rn "D_MWB\|visual.refactor" src/` returned no results in either tree). If a `D MWB` lane later emerges and reskins `CoachWorkoutBuilderScreen`, the inline exercise search section (`lines 1621–1657`) is the most likely collision surface with EW1b (mobile picker UI overhaul). Recommendation: flag any D MWB PR that touches `CoachWorkoutBuilderScreen.tsx` as requiring EW1b review.

### 4.5 Two-Catalog Coexistence Risk

The mobile layer currently has **two separate API clients** speaking to two separate backend modules:
- `exerciseLibraryApi.ts` → `GET /exercises/*` (Module A, ExerciseDB) — used by `CoachWorkoutBuilderScreen`
- `exerciseCatalog.ts` → `GET /exercise-catalog/*` (Module B, owner-curated) — used by `ExerciseLibraryScreen`, `ExerciseDetailScreen`

The `Exercise` shapes are **different** (`exerciseLibraryApi.Exercise` has `bodyPart`, `target`, `secondaryMuscles` as string[]; `exerciseCatalog.Exercise` has `primaryMuscle`, `category`, `equipment` as string[]). Any EW1 work that unifies the coach picker to use Module B must either create a mapper or produce a new unified type. Passing Module B results to the existing `addExercise(ex: Exercise)` without mapping would silently store `primaryMuscle` in `exercise_external_id` instead of the actual id.

## 6. Open Questions for Operator

**Q1 — Custom exercise visibility model (blocks EW1c schema)**
Should a coach-authored custom exercise be visible to other coaches in the same organisation (`org_shared`), or locked to the creator (`owner_only`) by default? The answer determines whether the `CustomExercise` table needs a `visibility` column and a `coach_id` tenant FK (joining to `User.coach_id`) — both affect the migration, the RLS policy, and the `ExerciseCatalogService` merge query. Affects EW1c spec directly.

**Q2 — Two-catalog unification strategy (blocks EW1a / EW1b scope)**
Does EW1 retire the ExerciseDB proxy (`GET /exercises/*`) in favour of the owner-curated catalog (`GET /exercise-catalog`), or does it produce a unified endpoint that returns merged results from both sources? Option A (retire ExerciseDB proxy) simplifies the shape but requires populating the catalog table with ~1,400 ExerciseDB entries (the seed script covers ~50). Option B (merge) preserves ExerciseDB's breadth but requires a type mapper on both client and server. This decision gates EW1a's endpoint design and EW1b's type adapter.

**Q3 — Difficulty taxonomy (blocks EW1a filter)**
`ExerciseCatalogItem.difficulty` is free-text `DEFAULT 'beginner'`. Should EW1 normalise this to an enum (`beginner | intermediate | advanced`) before adding a difficulty filter? Or keep free-text? Enum requires a backfill migration; free-text filter is case-sensitive unless handled with ILIKE. Affects the EW1a migration.

**Q4 — Video preview in the coach picker (blocks EW1b scope)**
Should the coach exercise picker support inline video preview while the picker is open (i.e., play the Mux HLS clip without navigating to `ExerciseDetailScreen`)? If yes, EW1b needs `expo-video` inside the modal and must handle the signed-URL lifecycle for the picker context. If no, tapping "Preview" pushes to `ExerciseDetailScreen` as a modal. The inline option is more complex and requires access to a `playbackUrl` inside the list response (currently only returned by the detail endpoint).

**Q5 — Secondary muscle filter UX (blocks EW1a/EW1b)**
Should the mobile picker expose secondary muscle as a separate filter chip row, or fold it into a combined "Muscles" multi-select that checks both `primary_muscle` and `secondary_muscles`? The combined approach requires an `OR` query on two columns (no single index covers both). A separate chip matches the current backend parameter model but adds a third muscle-related row to the filter UI.

**Q6 — Coach-side exercise library browse entry point**
Should EW1 add a standalone "Exercise Library" entry to the CoachNavigator (a dedicated browse screen the coach accesses outside the workout builder), or is the library exclusively accessed via the in-builder picker modal? The standalone entry costs a new route registration and a nav affordance (a toolbar button or a tab). Without it, coaches cannot bookmark exercises without opening a plan.

**Q7 — Telemetry event vocabulary for EW1**
What event names should EW1 instrument? Candidates: `exercise_library_opened`, `exercise_searched`, `exercise_filter_applied`, `exercise_detail_opened`, `exercise_video_played`, `exercise_added_to_plan`, `exercise_favorited`. Each needs a schema decision (properties, types) before the mobile telemetry wiring is built. Confirm PostHog as the sink (it's the connected PostHog connector in this workspace) and the event namespace convention.
