# R80-CLARIFY-1 — Executable code is the authoritative envelope, not comments

- **Ruling ID:** R80-CLARIFY-1
- **Date:** 2026-07-07
- **Operator:** Bradley Gleave <bradley@bradleytgpcoaching.com>
- **Autonomous delegate:** Op 53, executing under operator standing order "did you research what a hyperscaler would do? once you do, execute autonymously"
- **Clarifies:** R80 ("`extractors/_interface.js` in tgp-importer-extension is the entity envelope source of truth")
- **Triggered by:** wave-2 audit finding on PR #501 (Lens A) — `_interface.js` is internally inconsistent between its own comment (`{source_id, payload}` with provenance nested inside `payload`) and its own executable `makeEntity()` (which returns `{sourceId, sourcePlatform, payload, capturedAt}` with provenance at entity top level).

## Ruling

Where R80 makes `extractors/_interface.js` the source of truth, "source of truth" means the **executable code** — specifically the shape emitted by `makeEntity()` — **not** any comment, docstring, or JSDoc block in the same file.

For PR #501 (`feat/scout-ingest-endpoint`) and any subsequent backend PR that consumes the importer envelope:

- **Authoritative envelope shape (from `makeEntity()`):**
  ```
  {
    sourceId:       string,           // camelCase — stable ID from source platform
    sourcePlatform: "auto:<hostname>",// camelCase — provenance at ENTITY top level
    capturedAt:     ISO8601 string,   // camelCase — provenance at ENTITY top level
    payload:        object            // source-native JSON body only; NO provenance nested inside
  }
  ```
- **Backend DTOs MUST match this shape verbatim.** No renaming to `source_id`/`snake_case` at the boundary. No moving `sourcePlatform`/`capturedAt` inside `payload`.
- The stale comment in `_interface.js` describing `{source_id, payload}` MUST be rewritten to match `makeEntity()`.

## Rationale

Hyperscaler and industry consensus is unambiguous when executable code and adjacent documentation disagree:

- **"Only the Code Tells the Truth"** — O'Reilly, *97 Things Every Programmer Should Know*: "The ultimate semantics of a program is given by the running code… To know what a program does, the source is ultimately all you can be sure of looking at… Even the most accurate requirements document does not tell the whole truth… Comments are not running code. They can be just as wrong as other forms of documentation."
- **Google AIP-140** (protobuf-based APIs): field definitions use `lower_snake_case` internally but are **mapped to camelCase in JSON** at the wire. This directly parallels the shape emitted by `makeEntity()` (camelCase at the wire) and confirms camelCase is the correct wire convention for JSON-native TypeScript/JavaScript importers.
- **Contract-first API development** (DevGuide.dev, 2025): the executable contract (schema, generated types, running code) is what clients rely on. A drifted comment is a defect in the comment, not a defect in the code.
- **Working software over comprehensive documentation** (Ayende): "When a difference arises between what the documentation says vs. what the code is doing, the team must decide which is incorrect and correct it… Keeping a tight loop here is especially important for public APIs and API specifications."

Additionally: the extension is a live, MV3 service-worker Chrome extension that already emits `makeEntity()` payloads. Any backend that consumes a different shape will 400 on every extension call in production. The comment can be updated in a single-line diff; the extension emitting code can not be retrofitted without a Chrome Web Store release cycle. Aligning to the code is the lower-risk, faster-to-ship, and hyperscaler-conforming direction.

## Implementation contract

Two coordinated fixes MUST land in the same wave:

**Fix A — PR #501 backend DTO** (repo: `growth-project-backend`):
- Rename DTO field `source_id` → `sourceId`.
- Move `sourcePlatform` and `capturedAt` from `payload` to the entity object top level.
- Remove the schema validation that required `payload.sourcePlatform`; the service now reads `entity.sourcePlatform` directly.
- Update all tests (`scout-ingest.controller.spec.ts`, `scout-ingest.service.spec.ts`, `scout-ingest.dto.spec.ts` if it exists) to construct fixtures matching `makeEntity()` output.
- Update the migration column names if any Prisma column was named `source_id` and is directly serialized from the DTO field; DB column names may remain `snake_case` per Postgres convention, but the DTO→column mapping MUST be explicit.

**Fix B — extension `_interface.js` comment cleanup** (repo: `tgp-importer-extension`):
- Rewrite the leading comment/docstring on `_interface.js` to describe the shape actually emitted by `makeEntity()`.
- Add a one-line note: "This shape is the authoritative envelope per R80-CLARIFY-1 (2026-07-07). Do not add per-comment aliases."

## Non-goals

- Not renaming Postgres columns from `snake_case` — DB-side naming is orthogonal and per-table conventional.
- Not requiring the extension to change `makeEntity()`. The extension shape is frozen; the backend adapts to it.
- Not creating a versioned envelope. If the envelope needs to evolve later, that would be a separate ruling and a `v2` shape, not a comment/code split.

## Verification

Auditor Lens A on any subsequent PR touching `/api/scout/*` MUST verify:

1. Extension `_interface.js` `makeEntity()` output can be POSTed verbatim to `/api/scout/ingest` and returns 2xx (not 400) under `whitelist:true, forbidNonWhitelisted:true` validation.
2. DTO field names are camelCase and match `makeEntity()`.
3. The `_interface.js` comment describes the actual shape (no `source_id` / no nested provenance references).

Any deviation is a P1 finding against R80-CLARIFY-1.
