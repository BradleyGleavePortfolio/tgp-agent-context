# R-DARK-1 — Dark-launched route response is uniform 404 before auth

- **Ruling ID:** R-DARK-1
- **Date:** 2026-07-07
- **Operator:** Bradley Gleave <bradley@bradleytgpcoaching.com>
- **Autonomous delegate:** Op 53 (this ruling), executing under operator standing order "did you research what a hyperscaler would do? once you do, execute autonymously"
- **Supersedes:** none
- **Related:** wave-2 audit findings on PRs #500, #501, #502 (`DARK_ROUTE_GUARD_ORDERING`)

## Ruling

Any HTTP route gated by a feature flag that is currently OFF **MUST return `404 Not Found` uniformly**, regardless of caller identity, request method, or auth status. It **MUST NOT** return `401`, `403`, or any other status that discloses the route's existence to callers who do not have the intended role.

Concretely for the current NestJS backend:

- The feature-flag gate **MUST run before any auth guard**. Implementation approach: implement the flag check as either (a) a **global exception filter** wired at bootstrap that intercepts requests to disabled routes and returns 404 before the guard chain fires, or (b) a **Nest middleware** registered via `configure(consumer)` on `AppModule` for the specific route paths.
- Controller-scoped `@UseGuards(FeatureFlagGuard)` alone is **not sufficient** and constitutes a doctrine violation of R-DARK-1 because global `APP_GUARD` guards (`JwtAuthGuard`, `RolesGuard`, `UserThrottlerGuard`) execute first and leak 401/403 for unauth/wrong-role callers.
- Guard docstrings/comments that claim "indistinguishable from an unmounted route" **MUST** be technically accurate. Any claim of dark-launch semantics that is not enforced by the actual guard chain order is itself a P1 audit finding.

## Rationale

Hyperscaler and standards-body consensus is that dark-launched or feature-gated endpoints should be indistinguishable from routes that do not exist. Specifically:

- **OWASP Web Security Testing Guide** (auth-check ordering): "Verify the authorization check runs *before* any branch that can reveal whether the resource exists, what state it's in, or any other resource-specific metadata. Returning 404 for 'not found', 400 for 'wrong state', and 401 for 'not authenticated' is itself a leak — an attacker enumerates resource IDs and learns states without ever passing the auth gate. **Recommended response shape: uniform 404 for everything an unprivileged caller should not see.**"
- **AWS API Gateway documentation** explicitly recommends mapping the default `INVALID_AUTHORIZATION_TOKEN` (401-shape) and `MISSING_AUTHENTICATION_TOKEN` responses for unrecognized routes to `404` via the `x-amazon-apigateway-gateway-responses` OpenAPI extension. AWS treats "401 on a route that doesn't exist" as a defect.
- **RFC 7231 §6.5.3** allows 403 for forbidden-for-reasons-unrelated-to-credentials, but this is orthogonal to the disclosure question. Even where 403 is semantically permissible, using 404 for a dark-launched route is a defense-in-depth improvement, not a semantic violation.

The additional cost is minimal: one global filter or middleware plus a small env-driven allowlist of currently-enabled routes. The benefit is that any future dark-launched endpoint automatically inherits the correct behavior without per-controller guard wiring.

## Implementation contract

- **Location of the check:** `src/common/feature-flag/feature-flag-not-found.filter.ts` (or middleware equivalent). Registered globally in `main.ts` **before** `useGlobalPipes` and before the global guards resolve.
- **Configuration source:** the same environment-variable pair each feature already uses (e.g. `FEATURE_SCOUT_INGEST`, `FEATURE_EXTENSION_PAIRING`). Filter reads the env at request time (not cached at bootstrap) so ops can toggle without redeploy.
- **Coverage:** all methods, all paths, all callers. GET, POST, PATCH, DELETE, OPTIONS — all return 404 when the flag is OFF. No response body beyond the standard Nest 404 shape.
- **Preserved behavior when flag is ON:** identical to today — the existing controller + auth guards + role guards + throttle + validation pipe all run.
- **Test contract:** each feature-gated route MUST have a spec that asserts:
  - Flag OFF, no auth header → 404 (not 401).
  - Flag OFF, valid non-coach Bearer → 404 (not 403).
  - Flag OFF, valid coach Bearer → 404 (not 200).
  - Flag ON, no auth → 401 (unchanged from pre-ruling behavior).
  - Flag ON, non-coach → 403.
  - Flag ON, coach → 200/2xx.

## Backport scope

This ruling applies retroactively to open PRs #500, #501, #502 (the currently-in-flight backend PRs). All three MUST land the global filter/middleware in the same wave, and the controller-scoped `FeatureFlagGuard`s MUST be removed (or downgraded to no-ops behind the global filter) to avoid double-checking.

Future PRs that introduce feature-flagged routes MUST use the global mechanism from day one.

## Non-goals

- Not requiring `Cache-Control: no-store` on 404 responses (already the Nest default).
- Not requiring uniform response body — the standard Nest `HttpException` 404 body is acceptable.
- Not addressing method-not-allowed vs not-found distinction; disabled routes are 404 regardless of method.

## Verification

Auditor Lens B on any subsequent PR MUST verify by request:

```
FEATURE_X=false → GET|POST|PATCH|DELETE /api/x/* returns 404 without an Authorization header
FEATURE_X=false → same routes return 404 with a non-coach Bearer
FEATURE_X=false → same routes return 404 with a coach Bearer
```

Any 401 or 403 in those three probes is a P1 finding against R-DARK-1.
