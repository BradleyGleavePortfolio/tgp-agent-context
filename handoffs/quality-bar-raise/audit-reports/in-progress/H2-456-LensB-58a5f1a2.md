### F-03 — migration-dry-run.yml: Reversibility Check Is Existence-Only (Promise vs Implementation Gap)
**Severity:** P1  
**Rule:** R100.33 (observability / no swallowed failures), R100.42 (no phantom-bug defenses / no false security)  
**File:Line:** `.github/workflows/migration-dry-run.yml:93–129`  

**Evidence:** The workflow header comment (lines 11–18) explicitly promises:

> "4. If a `down.sql` exists, apply forward → down → forward again and verify schema parity (pg_dump -s diff is empty)."

The actual `reversibility-check` job (lines 93–129) only does:

```bash
if [ -f "$dir/down.sql" ]; then
  echo "  OK: down.sql present (reversible)."
  continue
fi
```

It checks for **file existence only**. It does not execute `down.sql`, does not run forward→down→forward, and does not compare schema parity with `pg_dump -s`. A `down.sql` containing `-- placeholder` or an invalid SQL statement will pass this check.

**Why it's a problem:** The comment creates a false assurance. An engineer trusting the CI will believe rollback validity has been verified. The first time a migration goes wrong at 2 AM, the gap surfaces in production. This is exactly the R82 scenario the workflow was designed to prevent.

**Recommended fix:** Either implement the forward→down→forward + schema parity check in a separate job (requires Postgres service, which `forward-only` already spins up — reuse it), OR change the header comment to accurately say "we verify down.sql exists but do not execute it." The second option is weaker safety but at least eliminates the documentation lie.

### F-05 — dangerfile.js §5: Async TODO Check May Silently Never Execute
**Severity:** P2  
**Rule:** R100.36 (no swallowed errors / silent failures), R100.43 (zero dead code)  
**File:Line:** `dangerfile.js:65–79`  

**Evidence:**

```js
const addedTodoCount = (async () => {
  // ...
})();

addedTodoCount.then((n) => {
  if (n >= 5) warn(`...`);
});
```

Danger v12 runs the dangerfile as a CommonJS module. After `require()`-ing the file, it flushes collected messages synchronously (or with a single microtask tick). The `async IIFE + .then()` pattern schedules the warn as a microtask that may fire **after** Danger has already collected messages. The Danger v12 changelog does not document that arbitrary `.then()` chains are awaited before flush.

**Why it's a problem:** The TODO density check (§5) is the only guard against incremental comment debt accumulation. If it silently never fires, developers see no warning regardless of how many TODO/FIXME markers they add.

**Recommended fix:** Use top-level `await` (supported in Danger v12 `dangerfile.js` via ESM or `--experimental-vm-modules`), or convert to a synchronous implementation that uses `danger.git.modified_files` line counts without async `diffForFile` calls, or document explicitly that this check is best-effort.

### F-06 — setup-branch-protection.sh: Full PUT Is Destructive to Pre-Existing Settings
**Severity:** P2  
**Rule:** R100.40 (same-bug-everywhere / known failure mode), R1 (decacorn quality — Apple/Google document destructive ops)  
**File:Line:** `scripts/setup-branch-protection.sh:62–88` (PAYLOAD + curl)  

**Evidence:** The script uses `PUT /repos/{owner}/{repo}/branches/main/protection` which **replaces the entire branch protection configuration**. The script's own header says "Idempotent — running twice is fine." This is correct for idempotency (same PUT twice = same result) but misleading about destructiveness.

Settings not in the PAYLOAD that would be silently clobbered:
- `required_signatures` (if operator had enabled signed commits)
- Any custom `restrictions` (allowed push users/teams/apps) added via the GitHub UI
- Any `required_linear_history` state different from the script's `true`

**Why it's a problem:** An operator who has manually added required_signatures or custom push restrictions will lose them silently on script re-run. The header says "idempotent" without clarifying it's a full-replace, not a patch.

**Recommended fix:** Add a documentation comment explicitly stating: "WARNING: This script REPLACES the entire branch protection config via PUT. Any settings configured outside this script (e.g. required_signatures, push restrictions) will be clobbered. Review and add them to this script if needed." Alternatively, consider using `gh api` `PATCH` semantics by fetching current config, merging changes, and re-applying.

**Severity:** P3

**File:Line:** `dangerfile.js:14`

### F-18 — dangerfile.js: Conventional Commits Check Doesn't Detect `BREAKING CHANGE` Footer
**Severity:** P3  
**Rule:** R100.38 (comments explain correctly), Conventional Commits spec completeness  
**File:Line:** `dangerfile.js:36`  

**Evidence:**
```js
const titleLooksConventional = /^(feat|fix|perf|deps|revert|docs|refactor|test|build|ci|chore)(\([^)]+\))?!?:\s/.test(danger.github.pr.title);
```

The regex checks the PR **title** only, and supports `!` for breaking changes (e.g. `feat!:`). However, the Conventional Commits v1 spec also permits `BREAKING CHANGE: <description>` in the **commit body/footer** without the `!` in the title. A PR titled `feat: remove deprecated endpoint` with `BREAKING CHANGE: removed /v1/users` in the body would pass the title check but Danger would not surface a breaking-change warning.

Note: `release-please` v4 DOES correctly parse `BREAKING CHANGE` footers from commit messages, so the **version bump** will be correct. The gap is only in the **Danger risk warning** layer.

**Recommended fix:** Add a check:
```js
if (danger.github.pr.body.includes('BREAKING CHANGE:')) {
  warn('This PR contains a BREAKING CHANGE footer. Ensure the major version bump strategy is intentional and documented.');
}
```
