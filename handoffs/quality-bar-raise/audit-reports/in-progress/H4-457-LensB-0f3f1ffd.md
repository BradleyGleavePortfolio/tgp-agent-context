### F-06 · P2 · `beforeAll` is synchronous but fires an async Promise
**File:** `test/deploy-readiness.spec.ts:52,101,118`  
**Evidence:**
```typescript
beforeAll(() => {          // ← NOT async
  // ...
  const flipPromise = autoFlip(...);  // returns Promise<FlipResult>
  (global as Record<string, unknown>).__readinessFlipPromise = flipPromise;
  // beforeAll returns undefined; flipPromise is NOT awaited
});
```
**Why this is a problem:** Jest's `beforeAll` supports returning a Promise (or using `async/await`), but here the synchronous function returns `undefined`. Jest considers the setup complete when `undefined` is returned. The `flipPromise` continues resolving asynchronously. The test suite happens to work because the promise resolves quickly (< 1ms in dry-run mode) and the individual `it()` blocks that need its result properly `await` it. But:
1. This relies on timing coincidence, not explicit sequencing.
2. In a slow environment or with `mode=apply` (network call to `flyctl`), the `beforeAll` could complete and tests start before the flip calculation finishes.
3. Using `(global as ...)` to pass a Promise between lifecycle hooks is an unusual pattern that bypasses Jest's intended async lifecycle mechanism.

**Fix:** Make `beforeAll` async and await `flipPromise` directly:
```typescript
beforeAll(async () => {
  // ...
  const flipResult = await autoFlip({ switches: registry.switches, env: process.env, mode: flipMode });
  report.flips = flipResult.plans;
  // No need for global.__readinessFlipPromise
});
```
Then remove the `global.__readinessFlipPromise` pattern entirely and use `report.flips` in the test and afterAll.

### F-07 · P2 · `if (true)` dead conditional wrapper in spec
**File:** `test/deploy-readiness.spec.ts:155`  
**Evidence:**
```typescript
if (true /* always evaluate; only ASSERT under prod mode */) {
  it('(prod-mode only) every MUST_SET switch…', async () => { … });
  it('(prod-mode only) every imported provider…', () => { … });
}
```
**Why:** An `if (true)` is a dead conditional — it can never be false. Jest discovers `it()` calls statically (they are registered at module evaluation time), so this construct provides zero semantic value. The comment implies the author wanted a way to visually group the prod-mode tests; the correct pattern is a `describe` block. The `if (true)` wrapper may also confuse static analysis tools and future readers about whether the block can be disabled by changing a variable.  

**Fix:**
```typescript
describe('prod-mode gates', () => {
  it('every MUST_SET switch has a non-placeholder value', async () => { … });
  it('every imported provider has its required vars set', () => { … });
});
```

### F-10 · P2 · ENV_RULES regex in `env-discovery.ts` is fragile against property reordering
**File:** `test/prod-readiness/env-discovery.ts:53`  
**Evidence:**

### F-18 · P3 · Learning ledger fingerprint `file:line:token` is line-number-unstable
**File:** `test/prod-readiness/stub-scanner.ts:107`  
**Evidence:**
```typescript
const fingerprint = `${rel}:${i + 1}:${pat.token}`;
```
Fingerprint format: `src/foo.ts:42:STUB`. Any line insertion **before** line 42 in `foo.ts` invalidates the ledger entry (the underlying STUB moves to line 43 but the fingerprint still says `:42:`). The test would then flag a stale entry for a still-legitimate debt decision.

**Why:** This is the brief's explicit concern: "is the fingerprint stable enough across small unrelated edits?" Answer: **No**. A common refactor (adding a license header, an import, or a blank line) above a tracked STUB will invalidate every ledger entry in that file below the insertion point.

**Fix (content hash approach):** Replace line number with a short hash of the line content:
```typescript
const lineHash = Buffer.from(line.trim()).toString('base64').slice(0, 12);
const fingerprint = `${rel}:${pat.token}:${lineHash}`;
```
This makes the fingerprint stable under insertions above the line, while still unique per (file, token, content). Stale entries would only occur when the actual code changes, not when surrounding lines shift.

### Architecture (R100.14–20): Responsibilities cleanly separated?
**PASS.** The 8 scanner modules have clear single-purpose APIs:
- `registry-loader.ts` — parse/validate YAML
- `env-discovery.ts` — discover env vars from three sources
- `stub-scanner.ts` — find stub/placeholder markers in source
- `learning-ledger.ts` — load and classify known findings
- `provider-wiring.ts` — check SDK import + env var presence
- `reporter.ts` — render results (console/markdown)
- `operator-keys-generator.ts` — write OPERATOR_KEYS_NEEDED.md
- `auto-flipper.ts` — dry-run/apply switch flips

**Cross-module coupling:** `reporter.ts` imports types from all other modules but only as `import type` — no runtime coupling. `operator-keys-generator.ts` imports types from `registry-loader` and `provider-wiring` (correct). No module imports from another module's internals.

**Orchestrator (spec) doing too much:** The spec's `beforeAll` does substantial orchestration (8 calls, ledger filtering, unregistered computation, flip planning). This is acceptable for a test orchestrator but the global-promise pattern (F-06) is a architectural smell.
